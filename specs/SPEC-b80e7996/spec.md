# 機能仕様: Worktree一覧のエージェント状態アニメーション

**仕様ID**: `SPEC-b80e7996`
**作成日**: 2026-02-16
**ステータス**: ドラフト
**カテゴリ**: GUI / Agent Status

**依存仕様**:

- SPEC-861d8cdf（エージェント状態の可視化 — archive/TUI版）
- SPEC-c4e8f210（Worktree Cleanup GUI — FR-515/FR-516 アクティブエージェントインジケーター）
- SPEC-1b98b6d7（Hook実行でGUI増殖しない）

**入力**: ユーザー説明: "Worktree一覧でタブに存在するアクティブエージェントのWorktreeにインジケーターが表示されているが、Claude Codeの場合はHookでLLMが動作中のみアニメーションしてほしい。Codexの場合はHookがないので常時アニメーション。また、アクティブエージェントの場合に一文字分インデントされてしまっている。"

## 背景

SPEC-c4e8f210 FR-515/FR-516 で Sidebar と Cleanup モーダルにエージェントタブのアクティビティインジケーター（3本バー）が追加された。しかし現状の実装には以下の問題がある:

1. **アニメーション未実装**: `.agent-tab-bar` に `animation-delay` のみ設定され、`animation` プロパティ（keyframes 参照）が存在しない。バーは静止した3本線として表示される
2. **状態区別なし**: 「タブが開いているか」だけで判定しており、エージェントが実際にLLM推論中かどうかが反映されない。Stopped 状態でもアニメーション対象になってしまう
3. **インデント不整合**: `agent-tab-icon`（12px幅）がアクティブエージェントのブランチにのみ追加されるため、非アクティブブランチと比較してブランチ名の開始位置がずれる
4. **BranchInfo にステータス不足**: `is_agent_running` ブール値のみで、Running/WaitingInput/Stopped の3状態をフロントエンドに伝達できない

本仕様では、LLMがリアルタイムに動作していることを視覚的に把握できる2層構造のインジケーターを実装する。

## 用語定義

- **2層構造**: レイヤー1（静的アイコン: タブが存在する）＋ レイヤー2（アニメーション: LLMが動作中）
- **正確な状態検出**: Claude Code の Hook API 経由で取得した AgentStatus（Running/WaitingInput/Stopped）
- **推測的状態検出**: Codex/Gemini/OpenCode のペイン出力解析・プロセス生存確認・アイドルタイムアウトによる状態推測

## ユーザーシナリオとテスト *(必須)*

### ユーザーストーリー 1 - Claude Code のLLM動作中アニメーション (優先度: P0)

開発者として、Worktree一覧で Claude Code がLLM推論を実行中のブランチを一目で把握したい。

**独立したテスト**: Claude Code エージェントを起動し、タスク実行中とタスク完了後でインジケーターの表示が変化することを確認する。

**受け入れシナリオ**:

1. **前提条件** ブランチ `feature/x` に Claude Code タブが開いており AgentStatus が Running、**操作** Sidebar（Local）を表示、**期待結果** 該当ブランチ行の先頭にアニメーションするインジケーターが表示される
2. **前提条件** Claude Code が Stop イベントを発火して AgentStatus が Stopped に遷移、**操作** Sidebar（Local）を表示、**期待結果** アニメーションが停止し、静的なインジケーター（タブ存在を示す）のみが表示される
3. **前提条件** Claude Code がパーミッション確認待ち（WaitingInput）、**操作** Sidebar（Local）を表示、**期待結果** アニメーションが停止し、静的なインジケーターのみが表示される（WaitingInput はLLM動作中ではない）
4. **前提条件** Hook経由で AgentStatus が Running に更新される、**操作** Sidebar を観察、**期待結果** セッションファイル変更のfs監視により、数秒以内にインジケーターがアニメーション開始する

---

### ユーザーストーリー 2 - Codex/Gemini/OpenCode のLLM動作推測アニメーション (優先度: P1)

開発者として、Hook API を持たないエージェント（Codex, Gemini, OpenCode）でも、LLMが動作中である可能性があることを視覚的に確認したい。

**独立したテスト**: Codex エージェントを起動し、コマンド実行中とプロンプト待機中でインジケーターの表示が変化することを確認する。

**受け入れシナリオ**:

1. **前提条件** ブランチ `feature/y` に Codex タブが開いており、プロセスが生存していてアイドル60秒以内、**操作** Sidebar（Local）を表示、**期待結果** アニメーションするインジケーターが表示される（Running推測）
2. **前提条件** Codex のプロセスが終了（Stopped）、**操作** Sidebar を表示、**期待結果** アニメーションが停止し、静的インジケーターのみが表示される
3. **前提条件** Codex のペイン出力末尾にプロンプトパターン（`>`, `→` 等）がある、**操作** Sidebar を表示、**期待結果** 推測状態が WaitingInput となり、アニメーションが停止する
4. **前提条件** Codex が60秒以上ペイン出力なし、**操作** Sidebar を表示、**期待結果** 推測状態が Stopped となり、アニメーションが停止する

---

### ユーザーストーリー 3 - インデント整合 (優先度: P0)

開発者として、エージェントタブの有無にかかわらず、Worktree一覧のブランチ名の開始位置が揃っていてほしい。

**独立したテスト**: エージェントタブが開いているブランチと開いていないブランチが混在するWorktree一覧を表示し、ブランチ名の水平位置が全行で揃っていることを確認する。

**受け入れシナリオ**:

1. **前提条件** エージェントタブがあるブランチと無いブランチが混在、**操作** Sidebar（Local）を表示、**期待結果** 全ブランチ行でブランチ名の水平開始位置が同一である
2. **前提条件** エージェントタブが1つも開いていない、**操作** Sidebar（Local）を表示、**期待結果** 全ブランチ行のインジケーター予約幅が確保され、レイアウトがずれない

---

### ユーザーストーリー 4 - 状態同期の即時性 (優先度: P1)

開発者として、エージェントの状態変化（Running→Stopped等）が数秒以内にSidebarに反映されてほしい。

**独立したテスト**: Claude Code の Hook イベントを発火し、セッションファイルの更新からSidebarのインジケーター変化までの遅延を計測する。

**受け入れシナリオ**:

1. **前提条件** `~/.gwt/sessions/` のセッションファイルが Hook によって更新される、**操作** Sidebar を観察、**期待結果** fs監視により変更を検知し、ブランチ一覧が自動更新されてインジケーター状態が反映される
2. **前提条件** fs監視が何らかの理由で失敗、**操作** Sidebar を表示し続ける、**期待結果** ポーリングフォールバック（数秒間隔）により状態が遅延反映される

---

### ユーザーストーリー 5 - Cleanup モーダルのインジケーター (優先度: P1)

開発者として、Cleanup モーダルでもエージェントのLLM動作状態を確認したい。

**独立したテスト**: Cleanup モーダルを開き、Running状態のエージェントタブがあるWorktreeにアニメーションインジケーターが表示されることを確認する。

**受け入れシナリオ**:

1. **前提条件** Running 状態のエージェントタブがあるブランチが存在、**操作** Cleanup モーダルを開く、**期待結果** 該当行にアニメーションインジケーターが表示される
2. **前提条件** Stopped 状態のエージェントタブがあるブランチが存在、**操作** Cleanup モーダルを開く、**期待結果** 該当行に静的インジケーターのみが表示される

## エッジケース

- エージェントタブが開いているがセッションファイルが存在しない → 静的インジケーターのみ表示（agent_status は Unknown 扱い）
- セッションファイルの `last_activity_at` が60秒超過 → Stopped として扱い、アニメーション停止
- Remote ブランチ一覧ではインジケーターを表示しない（既存動作を維持）
- `prefers-reduced-motion` が有効な場合 → アニメーションを無効化し、静的な `@` フォールバックを表示（既存動作を維持）
- 複数エージェント（異なるツール）が同一ブランチで起動 → 最新のセッション情報を優先
- fs監視の debounce: Hook は高頻度で発火するため、500ms以内の連続変更はバッチ処理する

## 要件 *(必須)*

### 機能要件

#### インジケーター表示

- **FR-800**: Sidebar（Local）の全ブランチ行に、インジケーター用の固定幅スペース（12px）を予約し**なければならない**。エージェントタブの有無にかかわらず、ブランチ名の水平位置を揃えること
- **FR-801**: エージェントタブが開いているブランチには、予約スペース内に静的インジケーター（パルスドット）を表示し**なければならない**（レイヤー1: タブ存在）
- **FR-802**: AgentStatus が Running のブランチには、静的インジケーターに替えてアニメーションインジケーター（パルスドット点滅）を表示し**なければならない**（レイヤー2: LLM動作中）
- **FR-803**: AgentStatus が WaitingInput または Stopped のブランチでは、アニメーションを停止し、静的インジケーターのみを表示し**なければならない**
- **FR-804**: Cleanup モーダルのブランチ列にも同様の2層インジケーター（FR-801〜FR-803）を表示し**なければならない**

#### データモデル

- **FR-810**: `BranchInfo` インターフェースに `agent_status` フィールド（`"unknown" | "running" | "waiting_input" | "stopped"`）を追加し**なければならない**。既存の `is_agent_running` は後方互換のため残すこと
- **FR-811**: `WorktreeInfo` インターフェースにも同様に `agent_status` フィールドを追加し**なければならない**
- **FR-812**: Rust バックエンドの `list_branches` / `list_worktrees` コマンドは、セッションから `AgentStatus` を読み取り、応答に `agent_status` を含め**なければならない**

#### 状態同期

- **FR-820**: Tauri バックエンドは `~/.gwt/sessions/` ディレクトリをファイルシステム監視（notify crate）し、変更検知時にフロントエンドへ Tauri イベント（`agent-status-changed`）を emit し**なければならない**
- **FR-821**: フロントエンドは `agent-status-changed` イベント受信時にブランチ一覧を再取得し、インジケーター状態を更新し**なければならない**
- **FR-822**: fs監視のフォールバックとして、フロントエンドは一定間隔（5秒）でブランチ一覧を再取得し**なければならない**（`is_agent_running` の変化のみチェック）
- **FR-823**: fs監視イベントは 500ms の debounce を適用し、高頻度の Hook 発火によるリフレッシュの嵐を防止し**なければならない**

#### エージェント状態検出

- **FR-830**: Claude Code エージェントの状態は、Hook 経由でセッションに書き込まれた `AgentStatus` をそのまま使用し**なければならない**（正確な検出）
- **FR-831**: Codex/Gemini/OpenCode エージェントの状態は、以下のヒューリスティクスで推測し**なければならない**:
  - プロセスが終了している → Stopped
  - `last_activity_at` から60秒以上経過 → Stopped
  - ペイン出力末尾にプロンプトパターン（`>`, `→`, `Input:`, `$` 等）がある → WaitingInput
  - 上記いずれにも該当しない（プロセス生存＆最近の出力あり） → Running
- **FR-832**: Rust バックエンドにセッションのアイドルタイムアウト判定（60秒）を組み込み、ブランチ一覧取得時に自動で Stopped へ遷移させ**なければならない**

### 非機能要件

- **NFR-800**: インジケーターアニメーションは CSS animation で実装し、JavaScript タイマーを使用しては**ならない**
- **NFR-801**: fs監視は `notify` crate を使用し、macOS (FSEvents) / Linux (inotify) / Windows (ReadDirectoryChangesW) に対応する**こと**
- **NFR-802**: fs監視のリソース消費を最小限に抑え、`~/.gwt/sessions/` ディレクトリのみを監視対象とする**こと**
- **NFR-803**: ブランチ一覧の再取得は軽量（既存 `list_branches` コマンド）であり、5秒間隔のポーリングでパフォーマンス問題を起こしては**ならない**

## 制約と仮定

### 制約

- Hook は `gwt-tauri hook <Event>` という別プロセスで実行され、実行中の GUI インスタンスに直接 Tauri イベントを emit できない。セッションファイルの変更をfs監視で検知する方式を採用
- Codex/Gemini/OpenCode の状態検出は推測であり、精度は Claude Code の Hook ベースの検出より低い

### 仮定

- Claude Code 1.0.3+ の Hook API が利用可能
- `notify` crate (v7+) が Cargo ワークスペースで利用可能

## 成功基準 *(必須)*

- **SC-800**: Claude Code の Running 状態時にインジケーターがアニメーションし、Stopped/WaitingInput 時に静止する
- **SC-801**: Codex のプロセス終了時にアニメーションが停止する
- **SC-802**: 全ブランチ行でブランチ名の水平位置が揃っている（インデント不整合が解消）
- **SC-803**: Hook による状態変更が fs 監視経由で 3 秒以内に Sidebar に反映される
- **SC-804**: Cleanup モーダルでも同様の2層インジケーターが機能する

## 範囲外

- エージェント状態に応じたタブ内の表示変更（タブラベルの色変え等）
- WaitingInput 状態の専用表示（点滅や色変え） — 本仕様では静的インジケーターのみ
- ステータスバーへの集計表示（TUI版 SPEC-861d8cdf FR-104 の GUI 移植）
