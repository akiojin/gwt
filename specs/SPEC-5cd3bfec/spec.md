# 機能仕様: 設定画面のスクロールをタブ切り替えに変更

**仕様ID**: `SPEC-5cd3bfec`
**作成日**: 2026-02-18
**更新日**: 2026-02-18
**ステータス**: ドラフト
**カテゴリ**: GUI

**入力**: ユーザー説明: "設定画面が長すぎるので、スクロールをタブに変更したい"

## 背景

- 現在の設定画面（SettingsPanel.svelte）は4つのセクション（Appearance, Voice Input, MCP Bridge, Profiles）を `<details>` アコーディオンで縦に並べ、`overflow-y: auto` でスクロールする構成
- セクション数が多く画面が長いため、目的のセクションを探すのに手間がかかる
- 同リポジトリの GitSection.svelte では既にタブ切り替えUIが実装されており、統一的なUXパターンが確立されている

## ユーザーシナリオとテスト *(必須)*

### ユーザーストーリー 1 - タブで設定セクションを切り替える (優先度: P0)

ユーザーとして、設定画面をスクロールせずに目的のセクションにタブクリックで素早くアクセスしたい。

**独立したテスト**: 設定画面を開き、各タブをクリックして対応するセクションの内容のみが表示されることを確認する。

**受け入れシナリオ**:

1. **前提条件** 設定画面が開いている、**操作** "Voice Input" タブをクリック、**期待結果** Voice Input セクションの内容のみが表示され、他セクションは非表示
2. **前提条件** 設定画面が開いている、**操作** "MCP Bridge" タブをクリック、**期待結果** MCP Bridge セクションの内容のみが表示される
3. **前提条件** 設定画面が開いている、**操作** "Profiles" タブをクリック、**期待結果** Profiles セクションの内容のみが表示される

---

### ユーザーストーリー 2 - 初期表示で Appearance タブが選択されている (優先度: P0)

ユーザーとして、設定画面を開いたとき最初に Appearance タブが表示されていてほしい。

**独立したテスト**: 設定画面を開き、Appearance タブがアクティブ状態であり、その内容が表示されていることを確認する。

**受け入れシナリオ**:

1. **前提条件** メインエリアにいる、**操作** 設定画面を開く、**期待結果** Appearance タブがアクティブで内容が表示されている

---

### ユーザーストーリー 3 - タブ切り替え後に Save/Close が常に表示されている (優先度: P0)

ユーザーとして、どのタブを表示していても Save/Close ボタンにアクセスできる。

**独立したテスト**: 各タブに切り替えた後、フッター（Save/Close ボタン）が常に画面下部に固定表示されていることを確認する。

**受け入れシナリオ**:

1. **前提条件** 設定画面の任意のタブを表示、**操作** 画面下部を確認、**期待結果** Save / Close ボタンが見えている

## エッジケース

- セクション内の内容がビューポートより長い場合は、タブコンテンツ領域内でスクロールする
- ウィンドウ幅が狭い場合でもタブバーが折り返さず全タブが操作可能である

## 要件 *(必須)*

### 機能要件

- **FR-001**: settings-body 内の `<details>` アコーディオン構造をタブバー + タブコンテンツ構造に置き換える
- **FR-002**: タブは Appearance / Voice Input / MCP Bridge / Profiles の4つ
- **FR-003**: 一度に表示されるタブコンテンツは1つだけ
- **FR-004**: 初期表示時は Appearance タブがアクティブ
- **FR-005**: タブ切り替え時に入力中の値は破棄されない（同一コンポーネントインスタンスの状態を維持）
- **FR-006**: タブUIスタイルは GitSection.svelte の `.git-tabs` / `.git-tab-btn` パターンを踏襲する

### 非機能要件

- **NFR-001**: タブ切り替えは即座に行われる（遅延なし）
- **NFR-002**: 既存の settings-header（タイトル + 閉じるボタン）と settings-footer（Save/Close ボタン）のレイアウトに変更を加えない

## 制約と仮定

- バックエンド（Rust）側の変更は不要（フロントエンドのみの変更）
- 新規コンポーネントファイルは作成せず、SettingsPanel.svelte 内で完結する
- 既存のテスト（SettingsPanel.test.ts）のセレクタは `<details>` に依存している可能性があるため、テスト側も修正する
