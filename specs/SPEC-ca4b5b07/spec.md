# 機能仕様: Issue タブ — GitHub Issue 一覧・詳細・フルフロー

**仕様ID**: `SPEC-ca4b5b07`
**作成日**: 2026-02-17
**更新日**: 2026-02-17
**ステータス**: ドラフト
**カテゴリ**: GUI
**依存仕様**:

- なし（既存の Issue 関連バックエンドコマンドを活用）

**入力**: ユーザー説明: "GitメニューにIssue一覧を表示するタブを追加する。Issue詳細のGFM Markdownレンダリング、Issueからworktree作成までのフルフローを実現する"

## 背景

- gwt は Git worktree ベースの開発ツールだが、GitHub Issue の閲覧・操作はブラウザに依存している
- Issue から作業を開始する導線（Issue → ブランチ → worktree → ターミナル）がアプリ内に存在しない
- バックエンドには `fetch_github_issues` / `find_existing_issue_branch` / `check_gh_cli_status` コマンドが既に存在するが、対応するフロントエンド UI がない
- Issue 本文を表示するための Markdown レンダリング機能がフロントエンドに未導入

## ユーザーシナリオとテスト *(必須)*

### ユーザーストーリー 1 - Issue 一覧の表示 (優先度: P0)

開発者として、Git メニューから Issue 一覧タブを開き、リポジトリのオープン Issue を一覧確認したい。

**独立したテスト**: Git メニュー「Issues」クリック → メインエリアに Issue 一覧タブが開き、オープン Issue がリスト表示される

**受け入れシナリオ**:

1. **前提条件** gh CLI インストール・認証済み、**操作** Git メニュー → Issues をクリック、**期待結果** メインエリアに「Issues (N)」タブが開き、全オープン Issue が更新日降順で表示される
2. **前提条件** Issue 一覧タブが既に開いている、**操作** Git メニュー → Issues を再クリック、**期待結果** 新規タブは開かず既存タブにフォーカス移動（シングルトン）
3. **前提条件** Issue が 30 件以上存在、**操作** 一覧を末尾までスクロール、**期待結果** 次ページの Issue が自動追加取得される（無限スクロール）
4. **前提条件** gh CLI 未インストール、**操作** Git メニュー → Issues をクリック、**期待結果** エラーメッセージとインストールガイドが表示される

---

### ユーザーストーリー 2 - Issue 一覧のフィルタリング (優先度: P0)

開発者として、テキスト検索とラベルフィルタで目的の Issue を素早く見つけたい。

**独立したテスト**: 検索バーにテキスト入力 → Issue タイトルでフィルタ。ラベルクリック → そのラベルの Issue のみ表示

**受け入れシナリオ**:

1. **前提条件** Issue 一覧表示中、**操作** 検索バーに "auth" と入力、**期待結果** タイトルに "auth" を含む Issue のみ表示
2. **前提条件** Issue 一覧表示中、**操作** ラベル "bug" をクリック、**期待結果** "bug" ラベル付き Issue のみ表示
3. **前提条件** フィルタ適用中、**操作** フィルタをクリア、**期待結果** 全 Issue が再表示

---

### ユーザーストーリー 3 - open/closed トグル (優先度: P1)

開発者として、クローズ済み Issue も確認したい。

**独立したテスト**: open/closed トグル切替 → closed Issue が表示される

**受け入れシナリオ**:

1. **前提条件** Issue 一覧がデフォルト表示（open）、**操作** "Closed" トグルをクリック、**期待結果** closed Issue が表示され件数も更新
2. **前提条件** closed 表示中、**操作** "Open" トグルをクリック、**期待結果** open Issue に戻る

---

### ユーザーストーリー 4 - Issue 詳細の表示 (優先度: P0)

開発者として、Issue をクリックして詳細（本文・メタ情報）を GFM Markdown でレンダリングして確認したい。

**独立したテスト**: Issue 行クリック → 詳細ビューに遷移、Markdown レンダリングされた本文が表示

**受け入れシナリオ**:

1. **前提条件** Issue 一覧表示中、**操作** Issue 行をクリック、**期待結果** 詳細ビューに切替。タイトル・本文（GFM）・ラベル・アサイニー・マイルストーン・コメント数が表示
2. **前提条件** 詳細ビュー表示中、**操作** 戻るボタンをクリック、**期待結果** 一覧ビューに戻る（フィルタ状態保持）
3. **前提条件** `spec` ラベル付き Issue の詳細を表示、**操作** Issue をクリック、**期待結果** セクション解析ビュー（IssueSpecPanel 相当）が表示

---

### ユーザーストーリー 5 - Issue からフルフロー（worktree 作成） (優先度: P0)

開発者として、Issue 詳細から「Work on this」で AgentLaunchForm を開き、Issue 情報がプリフィルされた状態で worktree を作成したい。

**独立したテスト**: 「Work on this」クリック → AgentLaunchForm が Issue 情報でプリフィルされて開く

**受け入れシナリオ**:

1. **前提条件** Issue 詳細ビュー（紐づき worktree なし）、**操作** 「Work on this」クリック、**期待結果** AgentLaunchForm が New Branch モードで開き、ブランチ名が `{prefix}issue-{number}` でプリフィル。prefix はラベルから推定（bug → bugfix/, enhancement → feature/, デフォルト → feature/）
2. **前提条件** Issue に "bug" ラベル付き、**操作** 「Work on this」クリック、**期待結果** prefix が "bugfix/" でプリフィル
3. **前提条件** Issue に紐づく worktree が存在、**操作** 詳細ビュー表示、**期待結果** 「Work on this」の代わりに「Switch to worktree」ボタンが表示

---

### ユーザーストーリー 6 - GitHub で開く (優先度: P1)

開発者として、Issue 詳細から GitHub の Issue ページを外部ブラウザで開きたい。

**独立したテスト**: 「Open in GitHub」クリック → 外部ブラウザで Issue ページが開く

**受け入れシナリオ**:

1. **前提条件** Issue 詳細ビュー表示中、**操作** 「Open in GitHub」クリック、**期待結果** デフォルトブラウザで該当 Issue の GitHub ページが開く

---

### ユーザーストーリー 7 - 手動リフレッシュ (優先度: P1)

開発者として、Issue 一覧を手動リフレッシュして最新状態を取得したい。

**独立したテスト**: リフレッシュボタンクリック → Issue 一覧が再取得

**受け入れシナリオ**:

1. **前提条件** Issue 一覧表示中、**操作** リフレッシュボタンをクリック、**期待結果** ローディング表示後、最新の Issue 一覧に更新

## エッジケース

- gh CLI 未インストール / 未認証 → エラー画面 + インストールガイド
- リポジトリに Issue が 0 件 → 空状態メッセージ表示
- GitHub API レート制限到達 → エラーメッセージ表示
- ネットワーク切断中 → エラーメッセージ + リトライボタン
- Issue 本文が非常に長い（10,000 文字超） → スクロール可能な表示
- Issue 本文に画像リンクがある → img タグとして表示（外部 URL）
- ラベルが大量（20 個以上）にある Issue → ラベルを折り返し表示
- 無限スクロール中のフィルタ変更 → 取得済みデータをクリアして再取得

## 要件 *(必須)*

### 機能要件

- **FR-001**: Git メニューに「Issues」メニュー項目を追加する
- **FR-002**: メニュークリックでメインエリアに Issue タブを開く（シングルトン）
- **FR-003**: タブラベルは「Issues (N)」形式（N = 表示中の Issue 件数）
- **FR-004**: Issue 一覧に #番号・タイトル・ラベル（色付き）・アサイニーアバター・更新日時・コメント数・マイルストーン・紐づき worktree インジケーターを表示
- **FR-005**: 更新日降順で固定ソート
- **FR-006**: テキスト検索バーによるタイトルフィルタ
- **FR-007**: ラベルクリックによるラベルフィルタ
- **FR-008**: open/closed トグルボタン
- **FR-009**: 無限スクロールによるページネーション
- **FR-010**: 手動リフレッシュボタン
- **FR-011**: Issue クリックで一覧→詳細切替ナビゲーション
- **FR-012**: Issue 本文を GFM (GitHub Flavored Markdown) でレンダリング
- **FR-013**: Issue メタ情報（タイトル・ラベル・アサイニー・マイルストーン・コメント数・ステータス）をヘッダーに表示
- **FR-014**: `spec` ラベル付き Issue は IssueSpecPanel のセクション解析ビューで表示
- **FR-015**: 「Work on this」ボタン → AgentLaunchForm を Issue 情報から全項目プリフィルで開く
- **FR-016**: ブランチ prefix をラベルから推定（bug → bugfix/, enhancement/feature → feature/, hotfix → hotfix/, デフォルト → feature/）
- **FR-017**: 紐づき worktree 存在時は「Switch to worktree」ボタンに変更
- **FR-018**: 「Open in GitHub」ボタンで外部ブラウザに Issue ページを開く
- **FR-019**: gh CLI 未対応時のエラー表示 + インストールガイド
- **FR-020**: 紐づき worktree インジケーターは `find_existing_issue_branch` API で判定

### 非機能要件

- **NFR-001**: GFM Markdown レンダリングは XSS 安全（HTML サニタイズ必須）
- **NFR-002**: メインエリアの幅に関わらずレスポンシブに表示
- **NFR-003**: 自動テストが必須（バックエンド: cargo test、フロントエンド: vitest）

## 制約と仮定

- gh CLI がインストール・認証済みであることを前提とする（未対応時はガイド表示）
- GFM Markdown レンダリング用ライブラリの新規導入が必要
- 既存の `fetch_github_issues` コマンドの返却型を拡張する必要がある（body, assignees, label color 等の追加）
- GitHub API のレート制限（認証済み: 5000 req/h）の範囲内で動作する

## 成功基準 *(必須)*

- **SC-001**: Git メニュー → Issues → Issue 一覧が表示されること
- **SC-002**: Issue クリック → GFM Markdown で本文が正しくレンダリングされること
- **SC-003**: 「Work on this」→ AgentLaunchForm が Issue 情報でプリフィルされて開くこと
- **SC-004**: 検索・ラベルフィルタ・open/closed トグルが正しく動作すること
- **SC-005**: 無限スクロールで全 Issue を閲覧できること
- **SC-006**: gh CLI 未対応時にエラー + ガイドが表示されること
- **SC-007**: 全テスト（cargo test, vitest）がパスすること
