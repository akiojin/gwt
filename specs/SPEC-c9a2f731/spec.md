# 機能仕様: Version History 即時表示（永続キャッシュ + プリフェッチ）

**仕様ID**: `SPEC-c9a2f731`
**作成日**: 2026-02-18
**更新日**: 2026-02-18
**ステータス**: ドラフト
**カテゴリ**: GUI / パフォーマンス

**依存仕様**:

- SPEC-133bf64f（Version History 基本機能 — 既存実装）

**入力**: ユーザー説明: "Version Historyですが、即座に表示して欲しいです。"

## 背景

- Version History タブを開くと `list_project_versions` → 各バージョンの `get_project_version_history`（AI要約生成含む）が逐次実行される
- AI 要約生成は数秒かかるため、タブを開いた瞬間は空の "Generating summary..." 表示になる
- キャッシュは `AppState` のインメモリのみで、アプリ再起動で消失する
- ユーザーが求めるのは「タブを開いた瞬間に情報が見えている」体験（100ms 以内）

## ユーザーシナリオとテスト *(必須)*

### ユーザーストーリー 1 - キャッシュ済みの即時表示 (優先度: P0)

ユーザーとして、Version History タブを開いた瞬間に、前回生成済みの AI 要約と changelog が即座に表示されてほしい。

**独立したテスト**: 永続キャッシュが存在する状態でタブを開き、100ms 以内にコンテンツが描画される

**受け入れシナリオ**:

1. **前提条件** 永続キャッシュに v1.0.0 の AI 要約 + changelog が保存済み、**操作** Version History タブを開く、**期待結果** v1.0.0 の status が即座に "ok" で summary_markdown + changelog_markdown が表示される
2. **前提条件** 永続キャッシュに unreleased の AI 要約が保存済みかつ OID 不変、**操作** Version History タブを開く、**期待結果** unreleased が即座に "ok" で表示される
3. **前提条件** 永続キャッシュに unreleased のキャッシュがあるが HEAD OID が変化、**操作** Version History タブを開く、**期待結果** 古いキャッシュの changelog が即座に表示され、バックグラウンドで AI 要約が再生成される

---

### ユーザーストーリー 2 - プリフェッチによるキャッシュ温め (優先度: P0)

ユーザーとして、プロジェクトを開いた時点でバックグラウンドで Version History のキャッシュが温まっていてほしい。

**独立したテスト**: プロジェクトオープン後に Version History タブを開かずに待機し、~/.gwt/cache/ にキャッシュファイルが生成される

**受け入れシナリオ**:

1. **前提条件** キャッシュが空、AI 設定済み、**操作** プロジェクトを開く、**期待結果** バックグラウンドで `list_project_versions` → キャッシュのないバージョンに対して AI 要約生成が開始される（最大 3 並列）
2. **前提条件** 全バージョンのキャッシュが有効（OID 一致）、**操作** プロジェクトを開く、**期待結果** プリフェッチは何も実行しない（無駄な API コール回避）
3. **前提条件** v1.0.0 のキャッシュのみ存在、unreleased は未キャッシュ、**操作** プロジェクトを開く、**期待結果** unreleased のみバックグラウンド生成される

---

### ユーザーストーリー 3 - 永続キャッシュの保存と読み込み (優先度: P0)

ユーザーとして、アプリを再起動しても Version History のキャッシュが消えずに残っていてほしい。

**独立したテスト**: キャッシュ生成 → アプリ再起動 → タブを開くと即座に表示

**受け入れシナリオ**:

1. **前提条件** AI 要約が正常生成された、**操作** キャッシュファイルを確認、**期待結果** `~/.gwt/cache/version-history/{repo-hash}.json` に全バージョンの AI 要約 + changelog が JSON で保存されている
2. **前提条件** キャッシュファイルが存在、**操作** `get_project_version_history` を呼ぶ、**期待結果** ディスクキャッシュ → インメモリキャッシュの順で復元し、キャッシュヒットを返す

---

### ユーザーストーリー 4 - changelog の即時フォールバック (優先度: P1)

ユーザーとして、AI 要約がまだ生成されていなくても、git log ベースの changelog だけは即座に見たい。

**独立したテスト**: キャッシュ空の状態でタブを開き、changelog が先に表示され、AI 要約が後から差し込まれる

**受け入れシナリオ**:

1. **前提条件** キャッシュ空、AI 設定済み、**操作** Version History タブを開く、**期待結果** まず changelog_markdown が表示され（status: "generating"）、AI 要約完了後に summary_markdown が追加される
2. **前提条件** キャッシュ空、AI 未設定、**操作** Version History タブを開く、**期待結果** changelog_markdown のみ表示、status: "disabled"

---

### ユーザーストーリー 5 - 並列 AI 要約生成 (優先度: P1)

ユーザーとして、複数バージョンの AI 要約が同時に生成されて早く結果が揃ってほしい。

**独立したテスト**: 5 バージョンの AI 要約を生成開始し、最大 3 つが同時に処理される

**受け入れシナリオ**:

1. **前提条件** 5 バージョンが全て未キャッシュ、**操作** プリフェッチ開始、**期待結果** 同時に最大 3 バージョンの AI 要約生成が実行され、1 つ完了するたびに次が開始される
2. **前提条件** 並列生成中、**操作** 1 つの生成が完了、**期待結果** フロントエンドでそのバージョンの status バッジが "ok" に変化

## エッジケース

- キャッシュファイルが破損（不正 JSON）→ 無視して再生成
- キャッシュファイルのディスクフル → エラーをログに記録、インメモリキャッシュのみ使用
- 言語設定変更（en→ja）→ 既存キャッシュの language フィールドが不一致→再生成
- アプリ終了中にキャッシュ書き込み → 部分書き込みが発生しうるため、次回起動時に検証
- リポジトリが削除/移動された → キャッシュは残るが次回アクセス時に不一致で無効化
- unreleased のコミット数が 0 → changelog も AI 要約も生成しない

## 要件 *(必須)*

### 機能要件

- **FR-001**: `~/.gwt/cache/version-history/{repo-hash}.json` にリポジトリ単位でキャッシュを永続化する。repo-hash はリポジトリの canonical パスの SHA-256 先頭 16 文字
- **FR-002**: キャッシュエントリはバージョン ID をキー、値に label / range_from / range_to / range_from_oid / range_to_oid / commit_count / language / summary_markdown / changelog_markdown を持つ
- **FR-003**: `get_project_version_history` はディスクキャッシュ → インメモリキャッシュの順でヒットチェックを行う
- **FR-004**: タグ付きバージョンのキャッシュは OID + language が一致する限り永久有効
- **FR-005**: unreleased のキャッシュは Version History タブを開いたときに OID を比較し、変化があれば再生成する
- **FR-006**: プロジェクトオープン直後にバックグラウンドでプリフェッチを開始する。対象はキャッシュが存在しないまたは stale なバージョンのみ
- **FR-007**: AI 要約の並列生成は最大 3 並列とする（セマフォで制御）
- **FR-008**: AI 未生成時のフォールバックとして、changelog_markdown を先行表示する
- **FR-009**: changelog_markdown も永続キャッシュに含める
- **FR-010**: AI 要約生成完了時にフロントエンドのステータスバッジをリアルタイム更新する（既存の `project-version-history-updated` イベントを活用）

### 非機能要件

- **NFR-001**: キャッシュヒット時、Version History タブを開いてから 100ms 以内にコンテンツが描画される
- **NFR-002**: プリフェッチはプロジェクトオープンの初期化と並列実行し、他の起動処理をブロックしない
- **NFR-003**: キャッシュファイルの読み書きはメインスレッドをブロックしない（async/spawn_blocking）

## 制約と仮定

- 永続キャッシュの形式は JSON（人間が読める）
- キャッシュファイルのサイズは通常数十 KB〜数百 KB 程度（10 バージョン × 数 KB）
- AI API のレートリミットは外部制約として受け入れる（並列度 3 で十分保守的）
- `resolve_repo_path_for_project_root` が返すパスをリポジトリの canonical パスとして使用する

## 成功基準 *(必須)*

- **SC-001**: キャッシュが温まっている状態で Version History タブを開くと、100ms 以内に全バージョンの AI 要約 + changelog が表示される
- **SC-002**: アプリ再起動後もキャッシュが有効で、即時表示が維持される
- **SC-003**: プロジェクトオープン後、Version History タブを開く前にバックグラウンドでキャッシュが温まっている
- **SC-004**: AI 未設定環境でも changelog は即時表示される
