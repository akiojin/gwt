# 機能仕様: Launch Agent From Issue ブランチプレフィックスAI判定

**仕様ID**: `SPEC-a2f8e3b1`
**作成日**: 2026-02-20
**更新日**: 2026-02-20
**ステータス**: ドラフト
**カテゴリ**: GUI
**依存仕様**:

- SPEC-c6ba640a（From Issue 基本機能）

**入力**: ユーザー説明: "Launch AgentでFrom Issueから作成されるブランチが常にfeature/になる問題を修正し、AIでIssue内容を分析してプレフィックスを自動判定する"

## 背景

- 現在の From Issue フローでは、ブランチプレフィックスを GitHub Issue のラベル名で判定している（`bug` → `bugfix/`、`hotfix` → `hotfix/`、それ以外 → `feature/`）
- 多くの Issue にはラベルが付いていない、または `enhancement` や `documentation` のような非 bug/hotfix ラベルしかないため、実質的にほぼすべてのケースで `feature/` になってしまう
- Issue のタイトルや本文を読めば人間には明白な bugfix であっても、ラベルがなければ `feature/` になるのは不自然
- 既存の Manual タブには AI ブランチ名提案機能（`suggest_branch_names`）があるが、From Issue フローには接続されていない

## ユーザーシナリオとテスト *(必須)*

### ユーザーストーリー 1 - ラベルによるプレフィックス自動判定（既存動作の維持） (優先度: P0)

ユーザーとして、`bug` ラベルが付いた Issue を選択したとき、プレフィックスが `bugfix/` に自動設定されてほしい。

**独立したテスト**: `bug` ラベル付き Issue を選択し、プレフィックスが `bugfix/` になること

**受け入れシナリオ**:

1. **前提条件** From Issue タブが表示されている、**操作** `bug` ラベル付き Issue を選択、**期待結果** プレフィックスドロップダウンが `bugfix/` に設定される（AI呼び出しなし）
2. **前提条件** From Issue タブが表示されている、**操作** `hotfix` ラベル付き Issue を選択、**期待結果** プレフィックスドロップダウンが `hotfix/` に設定される（AI呼び出しなし）

---

### ユーザーストーリー 2 - ラベルなし Issue のAIプレフィックス判定 (優先度: P0)

ユーザーとして、`bug`/`hotfix` ラベルがない Issue を選択したとき、AIがタイトル・本文・ラベルを分析して適切なプレフィックスを提案してほしい。

**独立したテスト**: ラベルなし Issue を選択し、AIがプレフィックスを判定してドロップダウンに反映されること

**受け入れシナリオ**:

1. **前提条件** AI が設定済み、**操作** ラベルなしの「Login page crash on invalid email」という Issue を選択、**期待結果** スピナー表示後、`bugfix/` がドロップダウンにセットされる
2. **前提条件** AI が設定済み、**操作** ラベルなしの「Add dark mode support」という Issue を選択、**期待結果** スピナー表示後、`feature/` がドロップダウンにセットされる
3. **前提条件** AI が設定済み、**操作** `enhancement` ラベル付きの Issue を選択、**期待結果** `bug`/`hotfix` ラベルがないため AI に判定を委ね、結果がドロップダウンにセットされる

---

### ユーザーストーリー 3 - AI判定中のUIフィードバック (優先度: P0)

ユーザーとして、AI がプレフィックスを判定している間、処理中であることが視覚的にわかるようにしてほしい。

**独立したテスト**: Issue 選択直後にプレフィックスドロップダウンが空欄+スピナーになり、判定完了後に値がセットされること

**受け入れシナリオ**:

1. **前提条件** AI 判定が必要な Issue を選択、**操作** ドロップダウンの状態を確認、**期待結果** ドロップダウンが空欄になりスピナーが表示される
2. **前提条件** AI 判定が完了、**操作** ドロップダウンの状態を確認、**期待結果** スピナーが消えてAI判定結果のプレフィックスが選択状態になる

---

### ユーザーストーリー 4 - AI判定結果のユーザー上書き (優先度: P0)

ユーザーとして、AI が提案したプレフィックスに同意しない場合、ドロップダウンから自由に変更できるようにしてほしい。

**独立したテスト**: AI判定後にドロップダウンで別のプレフィックスを選択できること

**受け入れシナリオ**:

1. **前提条件** AI が `feature/` と判定した Issue が選択されている、**操作** ドロップダウンで `bugfix/` に変更、**期待結果** ブランチ名が `bugfix/issue-{number}` に更新される

---

### ユーザーストーリー 5 - AI判定失敗時のフォールバック (優先度: P0)

ユーザーとして、AIが利用不可（未設定・タイムアウト・レート制限等）の場合、プレフィックスを手動で選択できるようにしてほしい。

**独立したテスト**: AI が未設定の状態でラベルなし Issue を選択し、プレフィックスドロップダウンが未選択状態になること

**受け入れシナリオ**:

1. **前提条件** AI が未設定、**操作** ラベルなし Issue を選択、**期待結果** プレフィックスドロップダウンが未選択（空欄）状態になり、ユーザーが手動で選択できる
2. **前提条件** AI がタイムアウトまたはエラーを返す、**操作** ラベルなし Issue を選択、**期待結果** スピナーが消えてドロップダウンが未選択（空欄）状態になる
3. **前提条件** AI が不正な値（`fix` や `enhancement` 等、4種以外の値）を返す、**操作** 結果を受け取る、**期待結果** ドロップダウンが未選択（空欄）状態になる

---

### ユーザーストーリー 6 - Issue連続切り替え時のキャンセルとキャッシュ (優先度: P1)

ユーザーとして、Issue を素早く切り替えたとき、前のAIリクエストがキャンセルされ、一度判定済みの Issue を再選択したときは即座にプレフィックスが反映されてほしい。

**独立したテスト**: AI 判定中に別の Issue を選択したとき、前のリクエストが破棄され新しいリクエストが発行されること

**受け入れシナリオ**:

1. **前提条件** Issue A を選択してAI判定中、**操作** Issue B を選択、**期待結果** Issue A のリクエスト結果は無視され、Issue B の判定が開始される
2. **前提条件** Issue A の AI 判定が完了済み、**操作** Issue B を選択後、再び Issue A を選択、**期待結果** Issue A のプレフィックスがキャッシュから即座に反映される（API再呼び出しなし）

---

### ユーザーストーリー 7 - プレフィル経由のAI判定 (優先度: P1)

ユーザーとして、Issue 一覧の「Work on this」ボタン経由で Issue が選択された場合も、同じAI判定フローを通ってほしい。

**独立したテスト**: 「Work on this」ボタンで Issue を選択し、ラベルなしの場合にAI判定が発動すること

**受け入れシナリオ**:

1. **前提条件** `bug` ラベル付き Issue の「Work on this」ボタンを押す、**期待結果** プレフィックスが `bugfix/` にセットされる（AI呼び出しなし）
2. **前提条件** ラベルなし Issue の「Work on this」ボタンを押す、**期待結果** AI 判定が発動し、スピナー表示後にプレフィックスがセットされる

## エッジケース

- AI がレスポンスで `release/` を返した場合 → 有効なプレフィックスとして受け入れる
- Issue のタイトルが空の場合 → AI に本文のみ渡す（タイトル空欄は GitHub 上では発生しないが防御的に扱う）
- Issue の本文が 500 文字を超える場合 → 先頭 500 文字で切り詰めて AI に渡す
- Issue の本文が null/空の場合 → タイトルとラベルのみで AI に判定させる
- `bug` と `hotfix` の両方のラベルが付いている場合 → `hotfix/` を優先（より緊急度が高い）
- ネットワーク切断中の場合 → AI 判定失敗としてユーザー手動選択にフォールバック
- AI レスポンスに余計なテキスト（説明文等）が含まれる場合 → 4択の単語を抽出し、見つからなければパースエラーとして手動選択にフォールバック

## 要件 *(必須)*

### 機能要件

- **FR-001**: Issue 選択時のプレフィックス判定は「ラベル優先・AIフォールバック」方式とする。`bug` ラベル → `bugfix/`、`hotfix` ラベル → `hotfix/`、それ以外 → AI 判定
- **FR-002**: AI 判定はバックエンドに新規 Tauri コマンド `classify_issue_branch_prefix` を追加して実装する。既存の `suggest_branch_names` とは別系統とする
- **FR-003**: AI に渡すコンテキストは Issue のタイトル + ラベル一覧 + 本文（先頭 500 文字）とする。本文の切り詰めはバックエンド（Rust）側で行う
- **FR-004**: AI のレスポンスは `feature` / `bugfix` / `hotfix` / `release` の 4 択のいずれかをプレーンテキストで返すプロンプト設計とする。システムプロンプトは英語でハードコードする
- **FR-005**: AI が 4 択以外の値を返した場合、またはパースに失敗した場合は、プレフィックスドロップダウンを未選択状態にしてユーザーに手動選択させる
- **FR-006**: AI 未設定（`ai-not-configured`）の場合は即座にユーザー手動選択にフォールバックする（通知は表示しない。サイレントフォールバック）
- **FR-007**: AI 判定中はプレフィックスドロップダウンを空欄にしてスピナーを表示する
- **FR-008**: AI 判定結果はユーザーがドロップダウンから自由に変更可能とする
- **FR-009**: Issue を連続で切り替えた場合、前の AI リクエスト結果は無視する（リクエスト ID ベースでの結果棄却）
- **FR-010**: 一度 AI 判定が完了した Issue の結果はフロントエンド側でキャッシュし、再選択時は API 再呼び出しを行わない
- **FR-011**: プレフィル（`Work on this` ボタン）経由の Issue 選択も同じラベル優先・AI フォールバックフローを通る
- **FR-012**: AI モデルはユーザーが設定した既存の AI プロバイダー/モデルを使用する（`ProfilesConfig::resolve_active_ai_settings()`）
- **FR-013**: `bug` と `hotfix` の両方のラベルがある場合は `hotfix/` を優先する
- **FR-014**: プレフィックスが未選択（空欄）の状態では Launch ボタンをディスエーブルにする

### 非機能要件

- **NFR-001**: AI 判定は既存の `AIClient` のタイムアウト設定をそのまま利用する。タイムアウトやエラー発生時は手動選択にフォールバック
- **NFR-002**: キャッシュはフォーム表示中（AgentLaunchForm コンポーネントのライフサイクル）のみ保持し、フォームを閉じた時点で破棄する

## 制約と仮定

- AI プロバイダーの設定は既存の gwt 設定機構（`ProfilesConfig`）に従う
- From Issue タブで表示される Issue 情報（タイトル、ラベル、本文）は `fetch_github_issues` コマンドで取得済みのデータを利用する。本文が未取得の場合は追加の API 呼び出しが必要になる可能性がある
- プレフィックスの種類は現行の 4 種（`feature/` / `bugfix/` / `hotfix/` / `release/`）から変更しない
- AI ブランチ名提案機能（Manual タブの `suggest_branch_names`）とは別系統とし、相互に影響しない

## 成功基準 *(必須)*

- **SC-001**: `bug` ラベル付き Issue 選択時に `bugfix/` がセットされる（既存動作の維持）
- **SC-002**: ラベルなし Issue 選択時に AI が適切なプレフィックスを判定し、ドロップダウンに反映される
- **SC-003**: AI 判定中にスピナーが表示され、判定完了後にプレフィックスがセットされる
- **SC-004**: AI 未設定時にドロップダウンが未選択状態になり、ユーザーが手動選択できる
- **SC-005**: Issue 連続切り替え時に前のリクエスト結果が無視され、最新の選択のみが反映される
- **SC-006**: 判定済み Issue のキャッシュが機能し、再選択時に即座にプレフィックスが反映される
- **SC-007**: プレフィル経由でも同じ判定フローが動作する
