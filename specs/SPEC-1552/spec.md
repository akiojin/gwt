### 背景

現在の gwt は Docker コンテナ内でのエージェント起動をサポートしている。プロジェクト内の `docker-compose.yml`、`Dockerfile`、`devcontainer.json` を検出し、適切なコンテナ環境でエージェントを実行する。サービス選択、コンテナライフサイクル管理（再作成、ビルド、維持）、ポートディスカバリーを提供する。

Unity 移行にあたり、C# でコンテナ管理ロジックを再実装し、オーバーレイパネルでサービス選択 UI を提供する。

**優先度:** P1維持（MVP直後）として実装する。コア機能（Agent管理、ターミナル、Git/GitHub連携）が完成した直後に着手。MVP時点ではホストOS上でのAgent実行のみサポート。

### ユーザーシナリオ

| ID | シナリオ | 優先度 |
|----|---------|--------|
| US-1 | プロジェクトを開いた際に Docker/DevContainer 設定が自動検出される | P0 |
| US-2 | ユーザーが Docker サービスを選択してコンテナ内でエージェントを起動する | P0 |
| US-3 | コンテナのライフサイクル（作成・再作成・停止）を管理する | P1 |
| US-4 | DevContainer 設定に基づいてコンテナ環境が自動構成される | P1 |
| US-5 | ユーザーが DevContainer 設定済みプロジェクトを開き、エージェントがコンテナ内で実行される | P1 |
| US-6 | コンテナ起動失敗時にエラー通知が表示され、ホスト OS フォールバックが提案される | P1 |

#### US-1 詳細

1. ユーザーがプロジェクトを開く
2. システムがプロジェクトルートから以下を検索する:
   - `docker-compose.yml` / `docker-compose.yaml`
   - `Dockerfile`
   - `.devcontainer/devcontainer.json` / `.devcontainer.json`
3. 検出結果がプロジェクト情報パネルに表示される
4. Docker 設定が検出された場合、コンテナ内起動オプションが有効になる

#### US-2 詳細

1. ユーザーがエージェント起動時に「コンテナ内実行」を選択する
2. docker-compose の場合、サービス一覧がオーバーレイパネルに表示される
3. ユーザーがサービスを選択する
4. コンテナが起動（または既存コンテナに接続）される
5. エージェントがコンテナ内で作業を開始する

#### US-5 詳細

1. ユーザーが DevContainer 設定済みプロジェクトを開く
2. システムが `.devcontainer/devcontainer.json` を検出する
3. DevContainer 設定に基づいてコンテナが自動構成・起動される
4. エージェントがコンテナ内で実行される

#### US-6 詳細

1. コンテナ起動が失敗する（Docker デーモン未起動、イメージ取得失敗等）
2. エラー通知が HUD のコンソールウィンドウに表示される
3. 「ホスト OS で実行」フォールバックが提案される
4. ユーザーが承認するとホスト OS でエージェントが起動される

### 機能要件

| ID | 要件 | 関連US |
|----|------|--------|
| FR-001 | プロジェクト内の Docker/DevContainer 設定ファイルを検出する | US-1 |
| FR-002 | `docker-compose.yml`, `Dockerfile`, `devcontainer.json` をサポートする | US-1, US-4 |
| FR-003 | Docker サービス選択 UI をオーバーレイパネルで提供する | US-2 |
| FR-004 | コンテナライフサイクル（作成・再作成・ビルド・停止）を管理する | US-3 |
| FR-005 | コンテナ内でのエージェント起動をサポートする | US-2, US-5 |
| FR-006 | VContainer で `IDockerService` として DI 登録する | - |
| FR-007 | docker exec PTY 方式でコンテナ内に PTY を確立する（`docker exec -it` 経由）。既存の PTY 管理インターフェース（IPtyService）と同じインターフェースで操作する | US-2, US-5 |

### 非機能要件

| ID | 要件 |
|----|------|
| NFR-001 | コンテナ起動時間: 30秒以内（キャッシュ済みイメージ使用時） |
| NFR-002 | コンテナ内 PTY のレイテンシはホスト PTY と同等（差異 < 100ms） |

### 設計判断

| 項目 | 採用 | 不採用 | 理由 |
|------|------|--------|------|
| PTY 通信方式 | docker exec PTY 方式（`docker exec -it` 経由） | SSH 接続方式 | 既存の IPtyService インターフェースと統一でき、追加のSSHデーモン設定が不要 |

### 成功基準

| ID | 基準 | 検証方法 |
|----|------|----------|
| SC-001 | Docker 設定のあるプロジェクトでコンテナ内エージェント起動が動作する | 統合テスト: docker-compose プロジェクトでエージェント起動を確認 |
| SC-002 | DevContainer 設定を正しく検出・利用できる | ユニットテスト: devcontainer.json パース・検証 |
| SC-003 | コンテナライフサイクル操作が正しく動作する | 統合テスト: 作成→停止→再作成のサイクルを確認 |
| SC-004 | DevContainer 内でエージェント起動・対話が動作する | 統合テスト: DevContainer プロジェクトでのエージェント起動・対話確認 |
| SC-005 | コンテナ起動失敗時のフォールバックが正しく動作する | 統合テスト: Docker 未起動状態でのフォールバック動作確認 |
