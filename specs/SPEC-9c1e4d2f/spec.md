# 機能仕様: ネイティブメニュー無反応時の検知性強化と受信経路修復

**仕様ID**: `SPEC-9c1e4d2f`
**作成日**: 2026-02-18
**更新日**: 2026-02-18
**ステータス**: ドラフト
**カテゴリ**: GUI
**依存仕様**:

- SPEC-b7f7b9ad（メニュー操作）
- SPEC-MENUFIX（フォーカスウィンドウスコープ）

**入力**: ユーザー説明: "メニューが効かない。About も開かない。"

## 背景

- `File > New Window` は動作する一方で `About` を含む `menu-action` 依存項目が無反応になる事象が再発している。
- 現状フロントエンド側のメニューリスナー初期化失敗が握りつぶされるため、原因断面が観測しづらい。
- イベントリスナー用ACLが不足すると `plugin:event|listen not allowed by ACL` で初期化が失敗する。
- 配布バイナリと開発環境での差分切り分けを短時間で行えるログとエラー可視化が必要。

## ユーザーシナリオとテスト *(必須)*

### ユーザーストーリー 1 - メニュー無反応を即座に検知できる (優先度: P0)

ユーザーとして、メニュー操作が効かない場合に無言で失敗せず、問題箇所を特定できる状態にしたい。

**独立したテスト**: メニューリスナー初期化失敗を発生させ、フロントエンドでエラー表示とログ出力が行われることを確認する。

**受け入れシナリオ**:

1. **前提条件** `setupMenuActionListener` が失敗する、**操作** アプリを起動する、**期待結果** `appError` にメニュー初期化失敗が設定される
2. **前提条件** 同上、**操作** コンソールログを確認する、**期待結果** 失敗理由を含む `console.error` が出力される
3. **前提条件** `setupMenuActionListener` が成功する、**操作** アプリを起動する、**期待結果** リスナー登録成功ログが1回出力される

---

### ユーザーストーリー 2 - バックエンド送信断面を追跡できる (優先度: P1)

開発者として、メニューイベントがRust側で受信・送信されたかをログから追跡したい。

**独立したテスト**: `on_menu_event` および `emit_menu_action` のログ出力がメニュー操作ごとに生成されることを確認する。

**受け入れシナリオ**:

1. **前提条件** メニュー項目をクリック、**操作** バックエンドログを確認、**期待結果** `on_menu_event` 受信ログに `id` が記録される
2. **前提条件** `emit_menu_action` が実行される、**操作** バックエンドログを確認、**期待結果** 送信先labelとactionが記録される
3. **前提条件** ターゲットwindow未解決、**操作** バックエンドログを確認、**期待結果** send-skip理由がwarnで記録される

## エッジケース

- Tauriランタイム外（テスト/ブラウザ文脈）で `@tauri-apps/api/webviewWindow` import が失敗した場合でも、無言失敗しないこと。
- フォーカスウィンドウが解決できない瞬間でも、処理スキップ理由がログに残ること。
- エラー表示追加により既存の `appError` 表示UI挙動を壊さないこと。

## 要件 *(必須)*

### 機能要件

- **FR-900**: システムは、`setupMenuActionListener` 初期化失敗時に明示的なエラーを投げなければならない。
- **FR-901**: システムは、`App.svelte` でメニューリスナー初期化失敗を握りつぶしてはならない。
- **FR-902**: システムは、メニューリスナー初期化失敗時にユーザーが視認できる `appError` を設定しなければならない。
- **FR-903**: システムは、`on_menu_event` 受信時にメニューIDをログ出力しなければならない。
- **FR-904**: システムは、`emit_menu_action` 実行時にターゲットlabel/actionおよび送信結果をログ出力しなければならない。
- **FR-905**: システムは、グローバル `listen` へのフォールバックを追加してはならない。
- **FR-906**: システムは、`capabilities/default.json` に `core:event:default` を明示的に許可しなければならない。

### 非機能要件

- **NFR-900**: 追加ログはメニュー操作のデバッグに必要な最小限情報に留めること。
- **NFR-901**: 公開コマンドAPIおよび既存メニューID契約を変更しないこと。

## 制約と仮定

- 症状再現環境は macOS + 配布バイナリを主対象とする。
- `File > New Window` は既に動作しており、一次原因は `menu-action` 受信経路にあると仮定する。
- 修正は既存ファイルの改修を優先し、新規フォールバック経路は導入しない。

## 成功基準 *(必須)*

- **SC-900**: `gwt-gui/src/lib/menuAction.test.ts` で初期化失敗時エラー伝播のテストがパスする。
- **SC-901**: `cargo test -p gwt-tauri` で既存メニューマッピングテストが回帰しない。
- **SC-902**: 手動確認で `About` / `Preferences` / `Check for Updates` が1クリックで動作するか、少なくとも失敗断面をログで一意に特定できる。
- **SC-903**: `capabilities/default.json` が `core:event:default` を含むことをテストで検証できる。
