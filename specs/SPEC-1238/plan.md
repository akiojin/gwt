# 実装計画: SPEC-1238

## 概要

2層の修正でリリースとアセットのレースコンディションを解消する。

## 設計方針

### 層1: Release ワークフロー修正（根本対策）

GitHub Release をドラフトとして作成し、全アセットアップロード完了後にドラフトを解除する。
これにより `/releases/latest` API はアセット準備完了後にのみ新バージョンを返す。

### 層2: キャッシュTTL短縮（防御的対策）

アセット未到着のキャッシュに対して短いTTL（5分）を適用し、早期に再チェックできるようにする。

## 変更ファイル

| ファイル | 変更内容 |
|---|---|
| `.github/workflows/release.yml` | ドラフトリリース作成 → アセット後に公開 |
| `crates/gwt-core/src/update.rs` | `PENDING_ASSET_TTL` 定数・ヘルパー・TTL選択ロジック・テスト追加 |

## リスク

- **低**: ドラフトリリースが解除されずに残るリスク（ビルド全失敗時）→ 手動で解除可能
- **低**: 既存キャッシュファイルとの互換性 → `serde(default)` により問題なし
