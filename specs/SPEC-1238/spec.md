# 機能仕様: 自動更新のレースコンディション修正

**仕様ID**: `SPEC-1238`
**作成日**: 2026-02-25
**更新日**: 2026-02-25
**ステータス**: 承認済み
**カテゴリ**: Infrastructure / Update
**依存仕様**: なし

**入力**: ユーザー説明: "GitHub Release が即座に公開されるが、バイナリアセットは約10分後にアップロードされるため、自動更新が失敗する"

## 背景

- GitHub Release が `--draft=false` で即座に公開され、バイナリアセットはビルド完了後（約10分後）にアップロードされる
- この間にアプリがアップデートチェックすると、新バージョンは検出されるがダウンロード可能なアセットがない
- 結果が24時間キャッシュされるため、アセット到着後も回復しない

## ユーザーシナリオとテスト *(必須)*

### ユーザーストーリー 1 - リリース直後の更新チェック (優先度: P0)

ユーザーとして、リリース直後にアップデートチェックしても、アセットが準備されるまで更新可能と表示されないようにしたい。

**独立したテスト**: ドラフトリリースはlatest APIに表示されないことをワークフロー実行で確認

**受け入れシナリオ**:

1. **前提条件** リリースワークフロー実行中（ビルド中）、**操作** アプリがアップデートチェック、**期待結果** `/releases/latest` はドラフトリリースを返さないため、既存バージョンが最新と判定
2. **前提条件** リリースワークフロー完了（アセットアップロード済み）、**操作** アプリがアップデートチェック、**期待結果** ドラフト解除後の新バージョンとアセットが検出される

---

### ユーザーストーリー 2 - アセット未到着キャッシュの早期回復 (優先度: P1)

ユーザーとして、万が一アセットなしの状態がキャッシュされても、5分後に再チェックされて回復してほしい。

**独立したテスト**: アセットなしキャッシュのTTLが5分であることをユニットテストで確認

**受け入れシナリオ**:

1. **前提条件** キャッシュに新バージョンありだがアセットURLなし、**操作** 5分経過後にチェック、**期待結果** キャッシュが無効化されAPIから再取得
2. **前提条件** キャッシュに新バージョンありでアセットURLあり、**操作** 24時間未満でチェック、**期待結果** 通常の24時間TTLでキャッシュヒット

## エッジケース

- 全プラットフォームのビルドが失敗した場合: ドラフトリリースのまま残り、ユーザーには影響なし
- 一部プラットフォームのビルドが失敗した場合: `fail-fast: false` + `needs` によりupload-releaseがスキップされドラフトのまま
- 手動でドラフト解除された場合: 防御的キャッシュTTL短縮（5分）で再チェックし回復

## 要件 *(必須)*

### 機能要件

- **FR-001**: リリースワークフローはGitHub Releaseをドラフトとして作成し、全アセットアップロード後にドラフト解除する
- **FR-002**: アップデートキャッシュにおいて、新バージョンありだがアセットなしの場合は5分のTTLを使用する
- **FR-003**: アセットが存在するキャッシュの場合は従来の24時間TTLを使用する

### 非機能要件

- **NFR-001**: 既存のリリースワークフローの冪等性を維持する
- **NFR-002**: キャッシュTTL変更は後方互換性を維持する（既存キャッシュファイルがそのまま読める）

## 制約と仮定

- GitHub REST API の `/releases/latest` はドラフトリリースを返さない
- `softprops/action-gh-release` はドラフトリリースへのアセットアップロードをサポートする

## 成功基準 *(必須)*

- **SC-001**: `cargo test` で新規テストを含む全テストが通過する
- **SC-002**: `cargo clippy --all-targets --all-features -- -D warnings` で警告なし
- **SC-003**: 次回リリース時にワークフローが正常動作する（ドラフト作成→ビルド→アセット→公開）
