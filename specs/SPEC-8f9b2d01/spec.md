# 機能仕様: gwt 起動時に前回のWindowを復元する

**仕様ID**: `SPEC-8f9b2d01`
**作成日**: 2026-02-17
**更新日**: 2026-02-17
**ステータス**: 実装完了
**カテゴリ**: GUI / Window
**依存仕様**: `SPEC-4470704f`（gwt GUI マルチウィンドウ + Native Windowメニュー）, `SPEC-f466bc68`（エージェントタブ復元）
**入力**: ユーザー説明: "gwtを起動すると、前回開いていたWindowが開く。"

## 背景

- マルチウィンドウ運用では、アプリを再起動した際に以前開いていたプロジェクト用ウィンドウが消えるため、再作業の時間がかかる。
- 既存機能では `open_project` 自体は復元されるが、前回開いていたウィンドウ単位の復元は行われず、起動時にウィンドウ数を再生成できていなかった。
- 本仕様は「起動時に前回開いたWindowの復元」に限定し、ウィンドウ内でのタブ内容復元は既存仕様に委ねる。

## ユーザーシナリオとテスト *(必須)*

### ユーザーストーリー 1 - 起動時に前回のWindowを再開したい (優先度: P0)

開発者として、前回起動時に開いていたWindowを再起動後も再現し、画面をすぐ作業可能状態にしたい。

**独立したテスト**: 2つ以上のウィンドウを開き、それぞれで別プロジェクトを開いた状態でアプリを再起動し、保存済みウィンドウが再開することを確認できる。

**受け入れシナリオ**:

1. **前提条件** 同一ユーザー環境でWindow A/Bが開き、各Windowが別プロジェクトを開いている、**操作** gwtを終了して再起動する、**期待結果** 各保存Windowが順次復元され、対応プロジェクトが自動再読込される
2. **前提条件** 保存したうち1Windowが既に終了済み、**操作** gwt再起動、**期待結果** 新規作成済みウィンドウ分のみ復元される

---

### ユーザーストーリー 2 - Window復元データが壊れても壊れないようにしたい (優先度: P1)

開発者として、ローカル保存データや権限不備でもアプリがクラッシュせず、可能な範囲で復元に進んでほしい。

**独立したテスト**: localStorage破損（JSON不正）時に復元が無効化されても起動自体が継続し、既存操作に影響しないことを確認できる。

**受け入れシナリオ**:

1. **前提条件** localStorage上の `gwt.windowSessions.v1` が破損している、**操作** gwt起動、**期待結果** アプリは起動し、破損データの復元はスキップされる
2. **前提条件** localStorage API が使えない環境、**操作** gwt起動、**期待結果** Window復元は best-effort で無視され、クラッシュしない

---

### ユーザーストーリー 3 - 復元時の衝突を吸収したい (優先度: P2)

開発者として、起動タイミングで複数Windowから同時復元処理が走っても、不要な競合を起こさず1回で済ませたい。

**独立したテスト**: 複数Windowが同時起動した状態でも、同一バッチで`gwt.windowSessions.restoreLeader.v1`ロックが有効となり、Window新規起動が重複しないことを確認できる。

**受け入れシナリオ**:

1. **前提条件** 保存ウィンドウ数が2以上、**操作** 全Window起動、**期待結果** 1Windowだけが他のWindow復元を実行し、重複した同一Window作成が発生しない
2. **前提条件** ラベル衝突を起こしうる状態、**操作** 起動復元、**期待結果** 競合時はサフィックス付きラベルが採番され、復元結果が保存される

## エッジケース

- 保存データの重複ラベル: 後勝ちで更新（ラベル単位の最後の状態を保持）
- 復元対象ウィンドウが現在アクティブ: 自ウィンドウでは `open_project` のみ行い、再openはスキップ
- `open_project` 失敗（削除済み/移動済みパス）: 該当セッションを削除して起動継続
- localStorage不可: Window復元と保存は best-effort で無視
- Tauri API未利用（Web preview）: 復元を行わずクラッシュしない

## 要件 *(必須)*

### 機能要件

- **FR-001**: アプリ起動時に、前回保存した `gwt.windowSessions.v1` を読み取り、保存済みwindow labelごとにプロジェクト復元を試みなければならない
- **FR-002**: 前回開いていたWindowが同時起動している場合、再開対象は1回だけ実行されなければならない（復元リーダーの排他）
- **FR-003**: 保存中の各Windowは、起動時に自身のlabelに紐づくprojectPathを `open_project` で復元しなければならない
- **FR-004**: 復元成功時、該当windowのセッション情報を最新ラベルで保存しなければならない
- **FR-005**: プロジェクトを開いたときに `gwt.windowSessions.v1` を更新し、Close Project時は該当Windowのセッションを削除しなければならない

### 非機能要件

- **NFR-001**: Window復元・保存は non-blocking（UI阻害なし）で best-effort で実行されること
- **NFR-002**: 競合時は新規ラベルを安全に採番し、既存セッションを壊さないこと

## 制約と仮定

- 復元は `open_project` ベースで行い、タブ単位の厳密再現は別SPEC（`SPEC-f466bc68`）で担保する
- 永続化は `localStorage` のみ（DBは使わない）
- ウィンドウ閉じ/再起動をまたいだ label 再衝突は suffix で解決する

## 成功基準 *(必須)*

- **SC-001**: 保存済みWindow分を起動時に復元できる
- **SC-002**: 1復元セッションが失敗しても他セッションの復元が継続される
- **SC-003**: 復元リーダー制御で重複Openが発生せず、複数Window同時起動でも想定どおりに復元される
