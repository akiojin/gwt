### 背景

gwt は macOS、Windows、Linux 向けのスタンドアロンデスクトップアプリケーションとして配布される。Unity 移行に伴い、ビルドパイプライン（Tauri → Unity Standalone Build）、自動更新メカニズム、システム監視、ログ管理、バグレポート生成を再実装する必要がある。

現在は Tauri ビルド（`.dmg`/`.msi`/`.AppImage`）と GitHub Release による配布を行っている。Unity では Unity Build Automation またはカスタム CI/CD パイプラインでクロスプラットフォームビルドを生成する。

#### 既存コマンド（移植対象）

- `check_app_update` / `apply_app_update`
- `get_system_info` / `get_stats`
- `read_recent_logs`
- `get_report_system_info` / `detect_report_target`
- `create_github_issue` / `capture_screen_text`

#### 自動更新

> **SPEC 更新（2026-03-10）**: ~~Sparkle (macOS) / Squirrel (Windows)~~ → **GitHub Release API 自前実装**に変更。全プラットフォーム共通の自前更新チェッカーを実装する。

- インプレース自動更新方式を採用（ダウンロード → 自動適用 → 再起動）
- 手動ダウンロードのみの方式は不採用
- **GitHub Release API による自前更新チェッカー**: バージョン比較 → ダウンロード → バイナリ置換 → 再起動
- 全プラットフォーム（macOS / Windows / Linux）で共通ロジック
- ~~macOS は Sparkle、Windows は Squirrel/WinSparkle を Unity プラグインとして統合~~ **廃止**

#### オフライン

- 非対応（ネットワーク接続前提）
- ただし#1548で記載の通り、ネットワーク切断時はローカル機能のみ継続

#### クラッシュ復旧

- スタジオ自動復元（~/.gwt/ のデータからスタジオ状態を完全復元。PTY セッションは失われるがデスク配置・Issue マーカーは復元）

#### テスト戦略

- 全層テスト + CI（EditMode/PlayMode/統合テスト）。テストカバレッジ 90% 以上

#### 配布方式

- Unity スタンドアロンビルド（macOS .app/.dmg, Windows .exe/.msi, Linux AppImage）
- 初回リリースから全プラットフォーム対応（macOS / Windows / Linux）を同時リリース
- Unity のクロスプラットフォームビルドを活用

#### IL2CPP 互換性

> **SPEC 更新（2026-03-10）**: IL2CPP + VContainer の互換性検証を**最優先（即座に検証）**に昇格。検証失敗時は Mono 代替または DI フレームワーク変更で対応する。

- VContainer + IL2CPP の互換性を**即座に検証（最優先タスク）**
- link.xml でのReflection保護、Source Generator活用を検討
- **検証失敗時の代替パス**: Monoバックエンドへのフォールバック、または VContainer から別 DI フレームワークへの変更

#### テレメトリ

> **SPEC 更新（2026-03-10）**: オプトインクラッシュレポート機能を追加。

- **オプトインクラッシュレポート**: ユーザー同意後にクラッシュ情報を GitHub Issue として自動作成
- 初回起動時にオプトイン確認ダイアログを表示
- クラッシュログ + システム情報 + 再現手順（可能な範囲）を自動収集
- ユーザーが内容を確認・編集した上で送信（完全透過的）

#### バグレポート機能（report.rs 5コマンド移植）

- **画面キャプチャ**: Unity Screenshot API + テキスト直接取得による画面キャプチャ機能
- **システム情報取得**: SystemInfo クラスによるシステム情報取得
- **ログ読み取り**: ~/.gwt/logs/ からの最近のログ読み取り
- **送信先検出**: バグレポート送信先の自動検出
- **Issue 送信**: GitHub Issue としてのバグレポート送信（gh CLI 経由）

### ユーザーシナリオ

| ID | シナリオ | 優先度 |
|----|---------|--------|
| US-1 | アプリ起動時に自動更新チェックが行われ、新バージョンがあれば通知される | P0 |
| US-2 | ユーザーが更新を適用し、アプリが再起動される | P0 |
| US-3 | HUD にシステムリソース（CPU、メモリ、GPU）が表示される | P1 |
| US-4 | ユーザーがバグレポートを作成し、システム情報付きで GitHub Issue が生成される | P1 |
| US-5 | 3プラットフォーム向けのビルドが CI/CD で自動生成される | P0 |
| US-6 | クラッシュ発生時、オプトイン同意済みユーザーにはクラッシュレポート送信確認が表示され、承認後にGitHub Issueとして自動作成される | P1 |

#### US-1 詳細

1. アプリ起動時にバックグラウンドで **GitHub Release API** を呼び出し、最新バージョンと比較
2. 新バージョンが利用可能な場合、HUD に通知バッジが表示される
3. ユーザーが通知をクリックすると更新詳細（バージョン、変更内容）が表示される

#### US-4 詳細

1. ユーザーがメニューから「Report Bug」を選択する
2. Unity Screenshot API で画面キャプチャが取得される
3. SystemInfo クラスでシステム情報（OS、CPU、メモリ、GPU、gwt バージョン）が自動収集される
4. 最近のログ（`~/.gwt/logs/`）が添付される
5. バグレポート送信先が自動検出される
6. ユーザーが問題の説明を入力する
7. GitHub Issue が gh CLI 経由で自動作成される

#### US-6 詳細

1. アプリクラッシュ検出（次回起動時にクラッシュログを検出）
2. オプトイン同意済みの場合、クラッシュレポート送信確認ダイアログを表示
3. クラッシュログ + システム情報を自動収集し、ユーザーが内容を確認・編集可能
4. 承認後、GitHub Issue としてクラッシュレポートを自動作成

### 設計判断

| 項目 | 採用 | 不採用 | 理由 |
|------|------|--------|------|
| 自動更新方式 | **GitHub Release API 自前実装**（バージョン比較→ダウンロード→バイナリ置換→再起動） | ~~Sparkle/Squirrel~~、ダウンロードURL提示のみ | 全プラットフォーム共通ロジックで実装をシンプルに。Sparkle/Squirrel のプラットフォーム個別統合の複雑さを排除 |
| プラットフォーム対応 | 初回リリースから全プラットフォーム同時対応 | 段階的対応 | Unity のクロスプラットフォームビルドにより同時リリースが可能 |
| スクリプティングバックエンド | IL2CPP（Mono フォールバック） | Mono のみ | パフォーマンス向上。ただし VContainer 互換性の**即座の検証が最優先** |
| テレメトリ | **オプトインクラッシュレポート（GitHub Issue自動作成）** | サードパーティテレメトリサービス | ユーザー透過性を最優先。既存のGitHub Issue基盤を活用 |
