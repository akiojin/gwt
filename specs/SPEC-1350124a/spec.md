# 機能仕様: Windows シェル選択（Launch Agent / New Terminal）

**仕様ID**: `SPEC-1350124a`
**作成日**: 2026-02-19
**更新日**: 2026-02-19
**ステータス**: ドラフト
**カテゴリ**: GUI
**依存仕様**:

- SPEC-f490dded（シンプルターミナルタブ）

**入力**: ユーザー説明: "Launch Agentで、Windows環境ではPowerShell/WSL/CommandPromptのどれを利用するのかを選択できるようにしたい"

## 背景

- 現在の gwt は Windows 上で PowerShell を自動選択してエージェント／ターミナルを起動しており、ユーザーがシェルを選択する手段がない
- Windows ユーザーは WSL (Windows Subsystem for Linux) や Command Prompt を好む場合があり、シェル固定はワークフローのボトルネックとなっている
- 特に WSL 内で開発環境を構築しているユーザーは、WSL 上のツールチェーンでエージェントを起動したいニーズがある
- macOS/Linux では `$SHELL` の自動検出で十分なため、シェル選択 UI は Windows 限定とする
- 現在、素のターミナル起動はネイティブメニュー（Tools > New Terminal）のみで、サイドバーに専用ボタンがない

## ユーザーシナリオとテスト *(必須)*

### ユーザーストーリー 1 - Launch Agent でシェルを選択する (優先度: P0)

Windows ユーザーとして、Launch Agent フォームの Advanced Options でシェル（PowerShell / Command Prompt / WSL）を選択し、選択したシェル環境でエージェントを起動したい。

**独立したテスト**: Windows 環境で AgentLaunchForm の Advanced Options にシェル選択ドロップダウンが表示され、選択したシェルでエージェントが起動すること

**受け入れシナリオ**:

1. **前提条件** Windows 環境で PowerShell が利用可能、**操作** Launch Agent フォームの Advanced Options でシェルに "PowerShell" を選択し起動、**期待結果** PowerShell 経由でエージェントコマンドが実行される
2. **前提条件** Windows 環境で WSL が利用可能、**操作** シェルに "WSL" を選択し起動、**期待結果** WSL のインタラクティブシェルが起動し、プロンプト検出後にエージェントコマンドが実行される
3. **前提条件** Windows 環境で Command Prompt が利用可能、**操作** シェルに "Command Prompt" を選択し起動、**期待結果** cmd.exe 経由でエージェントコマンドが実行される
4. **前提条件** macOS/Linux 環境、**操作** Launch Agent フォームを開く、**期待結果** シェル選択ドロップダウンが表示されない

---

### ユーザーストーリー 2 - WorktreeSummaryPanel の New Terminal ボタン (優先度: P0)

ユーザーとして、サイドバーの WorktreeSummaryPanel にある New Terminal ボタンをクリックして、デフォルトシェルでターミナルを即座に起動したい。

**独立したテスト**: WorktreeSummaryPanel に New Terminal ボタンが表示され、クリックでターミナルタブが作成されること

**受け入れシナリオ**:

1. **前提条件** プロジェクトが開いておりブランチを選択中、**操作** WorktreeSummaryPanel の New Terminal ボタンをクリック、**期待結果** デフォルトシェルで新しいターミナルタブが作成され、選択中ブランチの worktree パスでシェルが起動する
2. **前提条件** macOS 環境、**操作** New Terminal ボタンをクリック、**期待結果** $SHELL でターミナルが起動する（シェル選択 UI は非表示）
3. **前提条件** Windows 環境でデフォルトシェルが WSL に設定済み、**操作** New Terminal ボタンをクリック、**期待結果** WSL のインタラクティブシェルが起動し、worktree パスが /mnt/c/... 形式に自動変換される

---

### ユーザーストーリー 3 - Settings でデフォルトシェルを設定する (優先度: P0)

Windows ユーザーとして、Settings 画面の Terminal タブでデフォルトシェルを設定し、Launch Agent と New Terminal の両方に適用したい。

**独立したテスト**: Settings の Terminal タブにシェル選択セクションが表示され、保存後に設定ファイルに反映されること

**受け入れシナリオ**:

1. **前提条件** Windows 環境、**操作** Settings > Terminal タブで Default Shell を "WSL" に変更して保存、**期待結果** 設定ファイルにデフォルトシェルが "wsl" として保存される
2. **前提条件** US3-1 の状態、**操作** New Terminal ボタンをクリック、**期待結果** WSL でターミナルが起動する
3. **前提条件** macOS 環境、**操作** Settings を開く、**期待結果** Terminal タブ自体が表示されない

---

### ユーザーストーリー 4 - 利用可能シェルの事前検出 (優先度: P0)

Windows ユーザーとして、環境にインストールされていないシェルが選択肢に表示されないことで、混乱なくシェルを選択したい。

**独立したテスト**: WSL 未インストール環境で "WSL" が選択肢に表示されないこと

**受け入れシナリオ**:

1. **前提条件** Windows 環境で WSL 未インストール、**操作** シェル選択ドロップダウンを展開、**期待結果** "WSL" が選択肢に表示されない
2. **前提条件** Windows 環境で PowerShell 7 がインストール済み、**操作** シェル選択ドロップダウンを展開、**期待結果** "PowerShell" の横にバージョン情報（例: "7.x"）がサブテキストで表示される
3. **前提条件** Windows 環境で PowerShell 7 未インストール・Windows PowerShell 5.1 のみ、**操作** シェル選択ドロップダウンを展開、**期待結果** "PowerShell" の横に "5.1" がサブテキストで表示される

---

### ユーザーストーリー 5 - Docker モード時のシェル選択無効化 (優先度: P1)

Windows ユーザーとして、Docker モードを選択した場合にシェル選択が無効状態になり、無関係であることが明示されていることを確認したい。

**独立したテスト**: Docker サービス選択時にシェルドロップダウンが操作不能になること

**受け入れシナリオ**:

1. **前提条件** Windows 環境で Docker サービスを選択中、**操作** Advanced Options のシェル選択を確認、**期待結果** シェルドロップダウンが操作不能状態で、コンテナ内のデフォルトシェルが使用されることを示すテキストが表示される

---

### ユーザーストーリー 6 - WSL でのパス自動変換 (優先度: P0)

Windows ユーザーとして、WSL 選択時に worktree パスが自動的に /mnt/{drive_letter}/... 形式に変換され、WSL 内で正しいディレクトリに移動したい。

**独立したテスト**: Windows パス `C:\Users\foo\repo` が WSL パス `/mnt/c/Users/foo/repo` に変換されること

**受け入れシナリオ**:

1. **前提条件** Windows パスが `C:\Users\foo\repo`、**操作** WSL でエージェントまたはターミナルを起動、**期待結果** 作業ディレクトリが `/mnt/c/Users/foo/repo` に変換される
2. **前提条件** Windows パスが `D:\projects\repo`、**操作** WSL でターミナルを起動、**期待結果** 作業ディレクトリが `/mnt/d/projects/repo` に変換される

---

### ユーザーストーリー 7 - WSL での環境変数ハイブリッド処理 (優先度: P1)

Windows ユーザーとして、WSL 起動時にベース環境変数は WSL 側に委任しつつ、アプリ固有の環境変数オーバーライドは適用されることを確認したい。

**独立したテスト**: WSL 起動時にアプリの Profile 環境変数オーバーライドがシェル内で有効であること

**受け入れシナリオ**:

1. **前提条件** WSL 選択で Profile に `FOO=bar` の環境変数オーバーライドが設定されている、**操作** Launch Agent を起動、**期待結果** WSL シェル内で `echo $FOO` が `bar` を返す
2. **前提条件** WSL 選択で環境変数オーバーライドなし、**操作** Launch Agent を起動、**期待結果** WSL のログインシェルが起動し、WSL 側の .bashrc/.zshrc の環境変数が適用される

## エッジケース

- デフォルトシェルに設定したシェルがアンインストールされた場合 → 起動時にエラーを表示し、利用可能シェル一覧を再取得して自動的にフォールバック
- WSL のデフォルトディストリビューションが未設定の場合 → ディストリビューションの列挙が空であれば WSL を選択肢から除外
- Command Prompt でのエスケープ問題 → 初回はシンプルなコマンド実行方式で実装し、問題が発生したら専用ビルダーを追加（段階的対応）
- 設定ファイルに存在しないシェル識別子が保存されている場合 → 自動検出順で最初に見つかったシェルにフォールバック
- WSL パス変換でネットワークドライブ（UNC パス）の場合 → サポート外としエラーメッセージを表示
- WSL 起動後にプロンプトが 3 秒以内に検出できない場合 → 非インタラクティブ方式にフォールバックする

## 要件 *(必須)*

### 機能要件

- **FR-001**: アプリ設定にターミナル設定セクションを追加し、デフォルトシェルの識別子（"powershell" / "cmd" / "wsl"）を永続化できること。未設定時は自動検出順とする
- **FR-002**: Windows 環境で利用可能なシェルの一覧（識別子、表示名、バージョン情報）を取得する機能を提供すること。macOS/Linux では空の結果を返すこと
- **FR-003**: エージェント起動リクエストにシェル指定パラメータを追加し、指定されたシェルで起動可能とすること
- **FR-004**: AgentLaunchForm の Advanced Options に、Windows 環境でのみ表示されるシェル選択ドロップダウンを追加すること。利用可能なシェルのみ選択肢に含めること
- **FR-005**: 素のシェル起動コマンドにシェル指定パラメータを追加し、指定されたシェルで PTY を起動可能とすること
- **FR-006**: WSL 選択時、Windows パス（ドライブレター付き）を WSL パス（`/mnt/{drive_letter}/...`）に自動変換すること
- **FR-007**: WSL 選択時、WSL のインタラクティブシェルを PTY として起動すること。Launch Agent では PTY 出力からプロンプトパターン（行末 `$`, `#`, `>` 等）を 3 秒以内に検出した後にエージェントコマンドを投入すること。3 秒以内にプロンプトが検出できない場合は非インタラクティブ方式にフォールバックすること
- **FR-008**: Docker モード選択時、シェル選択ドロップダウンを操作不能状態にし、コンテナのデフォルトシェルが使用される旨を表示すること
- **FR-009**: Settings 画面に新規「Terminal」タブを追加し、Default Shell 設定セクションを配置すること。macOS/Linux ではタブ自体を非表示とすること
- **FR-010**: WorktreeSummaryPanel 内の Launch Agent ボタンの横に New Terminal ボタン（`>_` Unicode シンボル）を追加すること（全 OS で表示）。クリックでデフォルトシェルのターミナルタブを起動すること
- **FR-011**: WSL 起動時の環境変数はハイブリッド方式とすること。ベース環境変数は WSL のログインシェルに委任し、アプリの Profile 環境変数オーバーライドのみ追加で適用すること
- **FR-012**: シェル選択の初回デフォルト値は自動検出順（PowerShell → Command Prompt → WSL）で最初に見つかったシェルとすること
- **FR-013**: PowerShell は UI 上で「PowerShell」として統一表示し、実際に使用されるバージョン（7.x / 5.1）をサブテキストで表示すること
- **FR-014**: AgentLaunchForm でシェルを変更した場合、その選択を次回起動時のデフォルトとして永続化すること（エージェント・モデル選択と同様の挙動）

### 非機能要件

- **NFR-001**: 利用可能シェル一覧の取得は 1 秒以内に完了すること
- **NFR-002**: WSL パス変換は高速なバイト操作で行うこと（正規表現を使用しない）
- **NFR-003**: シェル設定の変更は即座に反映されること（アプリ再起動不要）

## 制約と仮定

- WSL はデフォルトディストリビューション固定とする（ディストリビューション選択は将来対応）
- WSL の PTY 起動方式は実装時に検証して最適なものを採用する
- Command Prompt のコマンドラッパーは段階的に対応する（初回はシンプルな方式）
- macOS/Linux ではシェル選択 UI を完全に非表示とする
- WSL の PTY 起動はインタラクティブ方式を優先するが、PTY ライブラリが `wsl.exe` を正常にハンドルできない場合は非インタラクティブ方式（`wsl.exe -e bash -lc 'command'`）にフォールバックする

## 成功基準 *(必須)*

- **SC-001**: Windows 環境で PowerShell / Command Prompt / WSL の3種類のシェルでエージェントが起動できる
- **SC-002**: WSL 選択時に Windows パスが自動的に /mnt/ 形式に変換され、正しいディレクトリで起動する
- **SC-003**: Settings で設定したデフォルトシェルが Launch Agent と New Terminal の両方に適用される
- **SC-004**: 環境にインストールされていないシェルが選択肢に表示されない
- **SC-005**: WorktreeSummaryPanel の New Terminal ボタンが全 OS で動作し、ターミナルタブが起動する
- **SC-006**: Docker モード時にシェル選択が操作不能になる
