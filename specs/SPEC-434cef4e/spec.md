# バグ修正仕様: v7.11.0 起動不能（Issue #1219）

**仕様ID**: `SPEC-434cef4e`
**作成日**: 2026-02-24
**更新日**: 2026-02-24
**ステータス**: ドラフト
**カテゴリ**: GUI
**依存仕様**:

- なし

**入力**: ユーザー説明: "Issue #1219: v7.11.0インストール後に起動できない問題を修正する"

## 背景

- v7.11.0 で `crates/gwt-tauri/src/bin/voice_eval.rs` を追加後、配布バンドルのメイン実行バイナリ選択が不安定になり、インストーラ成果物に `gwt` が同梱されず `voice_eval` のみが入るケースが発生した。
- この状態ではインストール後に GUI を起動できず、Windows/macOS の両方で致命的な利用不能状態になる。

## ユーザーシナリオとテスト *(必須)*

### ユーザーストーリー 1 - インストール後に gwt が起動できる (優先度: P0)

利用者として、配布版をインストールした直後に gwt を通常起動したい。

**独立したテスト**: リリース向けビルド成果物を確認し、起動用バイナリが `gwt` であることを検証する。

**受け入れシナリオ**:

1. **前提条件** macOS バンドルを生成済み、**操作** `*.app/Contents/MacOS` を確認、**期待結果** `gwt` 実行ファイルが存在する。
2. **前提条件** Windows バンドルを生成済み、**操作** Tauri ビルド対象を確認、**期待結果** `gwt-tauri`（GUI本体）が明示指定され `voice_eval` がメイン扱いにならない。

---

### ユーザーストーリー 2 - voice_eval を開発用途で継続利用できる (優先度: P1)

開発者として、GUI バンドル修正後も `voice_eval` ユーティリティを個別実行したい。

**独立したテスト**: Cargo メタデータで GUI メインバイナリと `voice_eval` が両立して定義されていることを確認する。

**受け入れシナリオ**:

1. **前提条件** `crates/gwt-tauri/Cargo.toml` を読み込む、**操作** バイナリ定義を確認、**期待結果** `gwt-tauri` と `voice_eval` の双方が明示される。

## エッジケース

- 将来 `src/bin/` に別ユーティリティを追加しても、リリースは常に `gwt-tauri` をメインとしてバンドルする。
- 開発用途 (`cargo run -p gwt-tauri --bin voice_eval`) の互換性を維持する。

## 要件 *(必須)*

### 機能要件

- **FR-001**: `crates/gwt-tauri` はメイン実行バイナリを `gwt-tauri` として明示しなければならない。
- **FR-002**: `voice_eval` は補助バイナリとして維持し、GUI バンドルのエントリポイントにしてはならない。
- **FR-003**: リリースワークフローは `cargo tauri build` 実行時に `gwt-tauri` バイナリを明示指定しなければならない。

### 非機能要件

- **NFR-001**: 修正は既存機能への影響を最小化し、差分はビルド設定とドキュメントに限定する。

## 制約と仮定

- 本修正は Issue #1219 の起動不能回避を優先し、`voice_eval` 自体の機能追加は行わない。
- 実機インストール検証は CI とローカル確認コマンドで代替する。

## 成功基準 *(必須)*

- **SC-001**: 生成された macOS バンドルに `gwt` 実行ファイルが含まれる。
- **SC-002**: リリースワークフロー定義上、Tauri ビルド対象が `gwt-tauri` に固定されている。
