# バグ修正仕様: Window復元時の無限生成防止

**仕様ID**: `SPEC-1f9d2a6c`  
**作成日**: 2026-02-17  
**更新日**: 2026-02-17  
**ステータス**: 確定  
**カテゴリ**: GUI  
**依存仕様**:

- [SPEC-4470704f](../SPEC-4470704f/spec.md)（マルチウィンドウ基盤）

**入力**: ユーザー説明: "Windowを復元すると同じWindowが無限に追加され続ける"

## 背景

- 起動時復元で、他ウィンドウ生成ロジックが連鎖し、Window数が増殖する不具合が発生した
- 復元リーダー制御がフロント `localStorage` 依存で、競合時に単一リーダー保証が崩れる余地があった
- 現在ウィンドウラベル取得で内部メタデータ fallback を使う経路があり、誤ラベル時に復元経路が誤作動する余地があった

## ユーザーシナリオとテスト *(必須)*

### ユーザーストーリー 1 - 起動時復元でWindowが増殖しない (優先度: P0)

ユーザーとして、アプリ再起動時に保存済みWindowが復元されても、同じWindowが無限に増えないことを保証したい。

**独立したテスト**: 複数Window状態を保存して再起動し、Window数が保存数を超えて増えないこと

**受け入れシナリオ**:

1. **前提条件** `main` と `project-*` の複数Windowセッションが保存済み、**操作** アプリを起動、**期待結果** 保存済みWindowのみが一度だけ復元され、連鎖生成しない
2. **前提条件** 復元対象が3Window以上、**操作** 連続で再起動を3回実行、**期待結果** 起動ごとにWindow数が指数的に増えない

---

### ユーザーストーリー 2 - 非リーダーWindowは他Windowを生成しない (優先度: P1)

ユーザーとして、`project-*` Windowが起動しても他Windowを再生成しないことを期待する。

**独立したテスト**: `project-*` ラベルで復元処理を実行しても `open_gwt_window` が呼ばれないこと

**受け入れシナリオ**:

1. **前提条件** 現在Windowラベルが `project-xxx`、**操作** 起動時復元処理を実行、**期待結果** 自ウィンドウのプロジェクト復元のみ実行される
2. **前提条件** `main` が復元リーダーを保持中、**操作** 別Windowの復元処理が走る、**期待結果** リーダー取得に失敗し他Window生成経路へ入らない

---

### ユーザーストーリー 3 - ラベル解決が誤っても復元が暴走しない (優先度: P1)

ユーザーとして、Windowラベル取得失敗時に危険な復元分岐へ進まないことを期待する。

**独立したテスト**: `get_current_window_label` 失敗時に復元処理が安全に中断されること

**受け入れシナリオ**:

1. **前提条件** `get_current_window_label` command が失敗、**操作** 起動、**期待結果** 復元処理は安全に中断し、Window増殖は発生しない

## エッジケース

- 復元リーダー解放 (`release`) が失敗した場合でも、次回起動でTTL満了後に再取得できること
- stale なリーダー情報が残っていても有効期限経過で新規取得できること
- 復元対象エントリに無効ラベル/空パスが混在しても無視して継続できること

## 要件 *(必須)*

### 機能要件

- **FR-001**: フロントエンドは現在Windowラベル解決に `get_current_window_label` command を使用しなければならない
- **FR-002**: 復元リーダー候補は `main` ラベルのWindowのみでなければならない
- **FR-003**: バックエンドは単一プロセス内で原子的な復元リーダーロック（TTL付き）を提供しなければならない
- **FR-004**: フロントエンドはリーダーロック取得成功時のみ他Window復元（`open_gwt_window`）を実行しなければならない
- **FR-005**: 非リーダーWindowは自Windowのプロジェクト復元のみを実行し、他Window生成を行ってはならない
- **FR-006**: 復元リーダーは `finally` で解放し、解放失敗時でも処理全体を失敗させてはならない
- **FR-007**: リーダーロック制御のユニットテスト（取得/拒否/期限切れ/解放）を追加しなければならない

### 非機能要件

- **NFR-001**: 起動時復元でWindow数が保存済み件数を超えて増殖しないこと（3回連続再起動で再現しない）
- **NFR-002**: 復元処理は best-effort を維持し、単一失敗でアプリ全体をクラッシュさせないこと

## 制約と仮定

- 復元リーダーTTLは既存運用値（15秒）を維持する
- UI文言やユーザー導線は変更せず、制御ロジックのみ修正する
- 保存済みWindowセッションのデータ形式（`gwt.windowSessions.v1`）は互換維持する

## 成功基準 *(必須)*

- **SC-001**: 無限Window増殖が再現手順で発生しない
- **SC-002**: `cargo clippy --all-targets --all-features -- -D warnings` が成功する
- **SC-003**: `cargo test -p gwt-tauri -- --nocapture` が成功する
- **SC-004**: `pnpm test src/lib/windowSessionRestoreLeader.test.ts` と `pnpm exec svelte-check --tsconfig ./tsconfig.json` が成功する
