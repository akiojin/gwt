### 背景

gwt Unity版は「カイロソフト ゲーム発展途上国」スタイルのトップダウン 3/4 ビュー開発スタジオとして実装される。
この「ゲーム」としての体験を高めるため、開発活動に基づくゲーミフィケーション機能を導入する。
ただし、コア機能（Git/ターミナル/エージェント管理）の完成後に実装する。

#### 概要

- 実績/バッジシステム
- スタジオ拡張（実績で新しい設備・装飾がアンロック）
- レベルシステム（プロジェクトにレベルがあり、開発活動で経験値が溜まる）
- エージェントのスキル成長（タスク完了でスキル XP 獲得）

#### 実装時期

- Phase 5（後回し）だが全て入れる予定

#### 既に決定している要素

- 雇用・解雇メタファー（エージェント、Lead）
- Issue マーカーの浮遊表現
- CI 状態による雰囲気変化（ライティング）
- スタジオのビジュアルフィードバック

#### 追加予定のゲーミフィケーション要素

- バッジ/実績システム（コミット数、PR マージ数、Issue 解決数等）
- スタジオレベル/拡張（実績に応じてスタジオの家具・装飾が増える）
- 実績トラッキング（開発活動の統計・可視化）
- エージェントのスキルごとの独立レベリング

#### 実績の対象

- **Git/GitHub活動**: コミット数、PRマージ数、Issue解決数、レビュー数
- **スタジオ運営活動**: Agent雇用数、セッション時間、Lead利用頻度
- 両方を組み合わせた実績システム

#### レベルシステム

- スタジオレベル = コミット数ベース
- **レベルカーブ**: 早めペース。Lv2 = 10コミット程度。上位レベルは指数的に増加。具体値はテストプレイで調整
- レベルアップで Agent 同時起動上限が増加（#1545 NFR-004 連動）-- 機能的要素
- レベルアップでスタジオ装飾・設備がアンロック -- ビジュアル要素
- `StudioLevel.GetMaxAgents()` による上限取得（装飾のみではなく機能的制約として連動）

#### Agent 同時起動上限テーブル（2026-03-10インタビュー確定）

レベル連動によるAgent同時起動上限。初期値として以下を設定し、テストプレイで調整する:

| スタジオレベル | Agent上限 |
|--------------|----------|
| Lv1 | 3 |
| Lv5 | 5 |
| Lv10 | 10 |
| Lv20 | 無制限 |

#### 家具・装飾（2026-03-10インタビュー確定）

- **家具は純粋に装飾**（機能的インタラクションなし）
- ゲーミフィケーションのアンロック対象として使用
- スタジオ装飾アンロックの具体的リストはテストプレイで決定

#### 推奨アセットスタック（Phase 5）

ゲーミフィケーション層の実装に以下のアセットを使用する（詳細は `docs/pixelart.md` 参照）:

| アセット | 価格 | 用途 |
|---------|------|------|
| SOAP (ScriptableObject Architecture Pattern) | $35 | ゲーム状態変数管理の基盤。SO変数・イベント・ランタイムセットで疎結合設計 |
| Shiny Stats | $18 | スタッフ（エージェント）のクラス・属性・レベル連動成長管理 |
| SBLS (Skill Based Leveling System) | $10 | スキルごとの独立レベリング（開発スキルの成長表現） |
| Easy Save 3 | $59 | ゲーミフィケーションデータの堅牢な永続化（AES暗号化対応） |
| Techtree / Skilltree Creator | $25 | 研究ツリー（新ジャンル・プラットフォーム解放）の UI |

**SOAP と VContainer の共存:**
- VContainer: DI 基盤（全サービスの依存注入）
- SOAP: ゲーム状態変数管理（ScriptableObject ベース、ゲーミフィケーション層専用）
- 両者は異なるレイヤーを担当し、共存する

**設計参考:**
- GameInc (GitHub, 無料): Game Dev Story/Game Dev Tycoon 風のアーキテクチャ参考
- BusinessTycoon (GitHub, 無料): テック企業経営シムの参考実装

#### データソース

- Git データ: コミット数、ブランチ数、マージ数（git log から取得）
- GitHub データ: PR マージ数、Issue 解決数、レビュー数（gh CLI から取得）
- gwt 独自データ: エージェント起動数、セッション時間、Project Mode 利用状況

### ユーザーシナリオ

| ID | シナリオ | 優先度 |
|----|---------|--------|
| US-1 | エージェントがタスク完了するとスキル XP を獲得し、レベルアップアニメーションが表示される | P2 |
| US-2 | 連続 CI green で実績バッジが解除され、通知が表示される | P2 |
| US-3 | 実績達成でスタジオの装飾がアンロックされる | P2 |
| US-4 | プロジェクトレベルが上がるとスタジオの設備が増え、Agent 同時起動上限が増加する | P2 |
| US-5 | 家具・装飾がアンロックされ、スタジオに配置される（純粋装飾、機能的インタラクションなし） | P2 |

#### US-1 詳細

1. エージェントがタスク（Issue 対応等）を完了する
2. タスクの種類に応じたスキル XP が付与される
3. XP が閾値に達するとスキルレベルがアップする
4. レベルアップアニメーション（エフェクト + 通知）が 2D スタジオ内で再生される

#### US-2 詳細

1. CI パイプラインが連続で green になる
2. 連続回数が実績閾値に達する（例: 5回連続、10回連続）
3. 実績バッジが解除される
4. スタジオ内エフェクト + HUD 通知が表示される

#### US-4 詳細

1. コミット数の蓄積によりスタジオレベルが上がる（Lv2 = 10コミット程度）
2. レベルに応じて Agent 同時起動上限が増加する（Lv1=3, Lv5=5, Lv10=10, Lv20=無制限）
3. レベルアップ通知が表示される
4. 新しいスタジオ装飾がアンロックされる

#### US-5 詳細

1. 実績やレベルアップにより家具・装飾がアンロックされる
2. ユーザーがアンロックされた家具をスタジオに配置する
3. 家具は純粋に装飾であり、機能的インタラクションは提供しない

### 機能要件

| ID | 要件 |
|----|------|
| FR-001 | 開発活動に基づく実績/バッジシステムを実装する（Git/GitHub活動 + スタジオ運営活動の両方を対象） |
| FR-002 | 実績達成でスタジオの視覚要素（設備・装飾）がアンロックされる |
| FR-003 | プロジェクトレベルシステム（経験値・レベルアップ）を実装する。レベル = コミット数ベース。レベルカーブは早めペース（Lv2 = 10コミット程度、上位は指数的増加） |
| FR-004 | Git/GitHub/gwt の活動データを実績トラッキングに使用する |
| FR-005 | ゲーミフィケーションデータを ~/.gwt/ に永続化する |
| FR-006 | VContainer で IGamificationService として DI 登録する |
| FR-007 | ゲーミフィケーション層のアーキテクチャ基盤として SOAP (ScriptableObject Architecture Pattern) を採用する |
| FR-008 | エージェントのスキル成長を Shiny Stats + SBLS で管理する |
| FR-009 | SOAP (ScriptableObject Architecture) でゲーム状態変数（スタジオレベル、実績ポイント等）を管理する |
| FR-010 | Shiny Stats でエージェントのクラス・属性・レベル連動成長を実装する |
| FR-011 | SBLS でスキルごとの独立レベリング（タスク完了でスキル XP 獲得）を実装する |
| FR-012 | バッジ・実績システム（初Issue close、初PR作成、連続CI green 等）を実装する |
| FR-013 | スタジオ拡張（デスク数増加、装飾アンロック）を実装する。家具・装飾は純粋に装飾（機能的インタラクションなし） |
| FR-014 | スタジオレベルと Agent 同時起動上限の連動を実装する（#1545 NFR-004 連動）。`StudioLevel.GetMaxAgents()` で上限を取得する。初期テーブル: Lv1=3, Lv5=5, Lv10=10, Lv20=無制限。装飾のみではなく機能的制約として連動する |
| FR-015 | スタジオ装飾アンロックシステムを実装する。具体的なアンロックリストはテストプレイで決定する |

### 非機能要件

| ID | 要件 |
|----|------|
| NFR-001 | ゲーミフィケーション機能はコア機能に影響しない（オプショナル） |
| NFR-002 | ゲーミフィケーションは開発のモチベーション向上が目的であり、作業の妨げにならない |
