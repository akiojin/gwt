### 背景

現在の gwt はプロジェクトファイルおよび GitHub Issues に対するセマンティック検索を提供している。ベクトル埋め込み（ChromaDB）を使用してセマンティックマッチングを実現し、ファイルインデックス、GitHub Issues インデックス、バックグラウンドインデックスをサポートする。

Unity 移行にあたり、C# でベクトル埋め込み・検索エンジンを再実装する。**SQLite-vec一本で検証を進める**。大規模プロジェクトでも高速にインデックス・検索できるようにする。

既存の SPEC #1520 が現行 Rust 実装をカバーしているが、Unity 再実装が必要。

#### ベクトルDB選定

- **採用: SQLite-vec** を唯一の候補として検証を進める
- Unity/C#環境での互換性は未検証であることを認識した上で、まずSQLite-vecで実装を進める
- 万が一SQLite-vecがUnity環境で動作しない場合は、その時点で代替ソリューション（FAISS、Hnswlib等）を検討する
- ChromaDB（現行Rust版）からの移行理由: 外部プロセス依存の排除、軽量化

#### 既存コマンド（移植対象）

- `ensure_index_runtime`
- `index_project_cmd`
- `search_project_index_cmd`
- `get_index_status_cmd`
- `index_github_issues_cmd`
- `search_github_issues_cmd`

### ユーザーシナリオ

| ID | シナリオ | 優先度 |
|----|---------|--------|
| US-1 | ユーザーがプロジェクトを開くとバックグラウンドでインデックスが自動作成される | P0 |
| US-2 | ユーザーが自然言語でプロジェクトファイルを検索する | P0 |
| US-3 | ユーザーが GitHub Issues をセマンティック検索する | P1 |
| US-4 | インデックスの進捗状況が確認できる | P1 |

#### US-1 詳細

1. ユーザーがプロジェクトを開く
2. バックグラウンドでファイルスキャンが開始される
3. 各ファイルの内容がベクトル埋め込みに変換される
4. 埋め込みがローカルデータベース（SQLite-vec）に保存される
5. 変更があったファイルのみ差分インデックスされる
6. HUD にインデックス進捗が表示される

#### US-2 詳細

1. ユーザーが検索バーに自然言語クエリを入力する（例:「ユーザー認証のロジック」）
2. クエリがベクトル埋め込みに変換される
3. 類似度の高いファイル・コードスニペットが検索結果に表示される
4. 検索結果をクリックするとファイルが開かれる

### 非機能要件

| ID | 要件 |
|----|------|
| NFR-001 | インデックス検索のレスポンスタイム: 100ms 以内（10万ファイル規模） |
| NFR-002 | インデックスサイズ: プロジェクトサイズの 10% 以下 |

### 成功基準

| ID | 基準 | 検証方法 |
|----|------|----------|
| SC-001 | プロジェクトファイルのセマンティック検索が動作する | 統合テスト: サンプルプロジェクトでインデックス作成 → クエリ → 期待結果の確認 |
| SC-002 | GitHub Issues のセマンティック検索が動作する | 統合テスト: テストリポジトリの Issues インデックス → クエリ → 結果確認 |
| SC-003 | バックグラウンドインデックスが非同期で完了する | ユニットテスト: インデックスタスクの非同期完了を確認 |
| SC-004 | 差分インデックスが正しく動作する | ユニットテスト: ファイル変更 → 変更分のみ再インデックスされることを確認 |
| SC-005 | ファイル検索・Issue 検索が100ms以内で結果を返す | パフォーマンステスト: 10万ファイル規模のプロジェクトでのレスポンスタイム計測 |
| SC-006 | インデックス再構築が5分以内に完了する | パフォーマンステスト: 10万ファイル規模のフルインデックス時間計測 |
