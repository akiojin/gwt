### アプローチ

1. `AssistantToolRegistry` trait を設計し、ツール定義の一括管理を実装する
2. 各ツールカテゴリを独立したハンドラとして実装する（疎結合）
3. Responses API (OpenAI互換) の tools 形式に合わせた JSON Schema 生成を実装する
4. ツール実行パイプライン（引数バリデーション → 権限チェック → 実行 → 結果構造化）を共通基盤として実装する

### 設計方針

- **ツール定義は静的**: 起動時に全ツールの JSON Schema を生成し、LLM セッション開始時に渡す。動的追加なし
- **権限は定義レベルで制御**: 危険な操作（force push）はツール定義自体を持たない。LLM が呼べない = 最も安全
- **エラーは構造体で返す**: 例外を投げず、`{ "error": true, "code": "...", "message": "..." }` を LLM に返す。LLM が自律リカバリ
- **async 一貫**: 全ハンドラは tokio ベース。UI スレッドブロックなし

### リスク

| リスク | 影響 | 対策 |
|--------|------|------|
| ツール数が多すぎて LLM コンテキストを圧迫 | ツール選択精度低下 | 40 個以下に制限、カテゴリ別にグルーピング |
| GitHub API レートリミット | SPEC 生成中に API 制限 | キャッシュ + 指数バックオフ |
| LLM が想定外のツール引数を渡す | 実行エラー | JSON Schema バリデーション + エラー構造体返却 |
