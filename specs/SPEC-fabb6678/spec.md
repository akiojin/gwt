# 機能仕様: 問題報告・改善提案機能（Error Reporting & Feature Suggestion）

**仕様ID**: `SPEC-fabb6678`
**作成日**: 2026-02-22
**更新日**: 2026-02-22
**ステータス**: 実装済み（OS ネイティブ画像キャプチャは Phase 2 として延期）
**カテゴリ**: GUI / Backend
**依存仕様**:

- SPEC-0e6abbb7（全画面テキストコピー Cmd+Shift+C — テキストキャプチャを報告に流用）

**入力**: ユーザー説明: "gwt で発生した問題を報告する機能、改善提案をする機能。問題が発生した場合にはスクショの内容を添付できる。UIも必要。"

## 背景

- gwt 利用中に問題が発生した場合、ユーザーが開発者に報告する手段がない。手動で GitHub Issues を開き、環境情報やエラー内容を転記する必要がある
- 改善提案も同様に、アプリ外で GitHub Issues を作成する必要があり、利用中の文脈を伝えにくい
- 現在のエラーハンドリングは各コンポーネントに散在しており（分散型 try/catch）、統一的なエラー検知・分類・通知の仕組みがない
- 既存の Cmd+Shift+C テキストキャプチャは AI への問題伝達用に設計されたが、GitHub Issues への直接報告には結びついていない

## ユーザーシナリオとテスト *(必須)*

### ユーザーストーリー 1 - エラー発生時の自動通知と問題報告 (優先度: P0)

ユーザーとして、gwt の操作中に予期しないエラーが発生した際、トースト通知で報告を促され、ワンクリックで問題報告フォームを開いてエラー情報を GitHub Issue として報告したい。

**独立したテスト**: Tauri コマンドが予期しないエラーを返した際に、トースト通知が表示され、"Report" リンクからモーダルが開くことを確認

**受け入れシナリオ**:

1. **前提条件** gwt が起動しプロジェクトが開いている、**操作** Tauri コマンドが severity=error 以上のエラーを返す、**期待結果** 画面下部にトースト通知が表示され "An error occurred. Report?" のメッセージと "Report" リンクが表示される
2. **前提条件** トースト通知が表示されている、**操作** "Report" リンクをクリック、**期待結果** 問題報告モーダルが開き、エラー情報（メッセージ・コマンド名・タイムスタンプ）がプリフィルされている
3. **前提条件** 同一セッション中に同じエラーが再度発生、**操作** エラーが発生する、**期待結果** 同一フィンガープリントのトースト通知は抑制される（再表示されない）
4. **前提条件** ブランチ名重複など severity=info のユーザー操作エラー、**操作** エラーが発生する、**期待結果** トースト通知は表示されない（既存のインラインエラー表示のみ）

---

### ユーザーストーリー 2 - メニューからの能動的な問題報告 (優先度: P0)

ユーザーとして、問題に気づいた際に Help メニューから問題報告フォームを開き、タイトル・再現手順・期待結果・実際の結果を入力し、診断情報とスクリーンキャプチャを添えて GitHub Issue を作成したい。

**独立したテスト**: Help メニューの "Report Issue..." から問題報告モーダルを開き、必要事項を入力して Issue を作成できることを確認

**受け入れシナリオ**:

1. **前提条件** gwt が起動している、**操作** Help メニュー → "Report Issue..." を選択（またはショートカット）、**期待結果** 問題報告モーダルが "Bug Report" タブで開く
2. **前提条件** 問題報告モーダルが開いている（Bug Report）、**操作** フォームを確認する、**期待結果** "Title"、"Steps to Reproduce"、"Expected Result"、"Actual Result" の入力フィールドが表示されている
3. **前提条件** フォームに必要事項を入力済み、**操作** "Submit" ボタンを押す、**期待結果** GitHub Issues に Issue が作成され、成功トーストが表示される。Issue には "bug" ラベルが付与されている

---

### ユーザーストーリー 3 - 改善提案の送信 (優先度: P0)

ユーザーとして、gwt に欲しい機能を改善提案として Help メニューから送信し、提案内容・ユースケース・期待される効果を GitHub Issue として起票したい。

**独立したテスト**: Help メニューの "Suggest Feature..." から改善提案モーダルを開き、Issue を作成できることを確認

**受け入れシナリオ**:

1. **前提条件** gwt が起動している、**操作** Help メニュー → "Suggest Feature..." を選択、**期待結果** 報告モーダルが "Feature Request" タブで開く
2. **前提条件** モーダルが Feature Request モードで表示されている、**操作** フォームを確認する、**期待結果** "Title"、"Description"、"Use Case"、"Expected Benefit" の入力フィールドが表示されている
3. **前提条件** フォームに必要事項を入力済み、**操作** "Submit" ボタンを押す、**期待結果** GitHub Issues に Issue が作成され、"enhancement" ラベルが付与されている

---

### ユーザーストーリー 4 - スクリーンキャプチャの添付 (優先度: P1)

ユーザーとして、問題報告時にテキストキャプチャ（既存 Cmd+Shift+C）または画像キャプチャ（OS ネイティブスクリーンショット）を選択して報告に添付したい。

**独立したテスト**: 報告モーダル内でテキストキャプチャと画像キャプチャの各ボタンを使い、プレビュー表示されることを確認

**受け入れシナリオ**:

1. **前提条件** 問題報告モーダルが開いている、**操作** "Capture Screen Text" ボタンをクリック、**期待結果** 既存の Cmd+Shift+C と同じ構造化テキストが取得され、プレビューエリアに表示される
2. **前提条件** 問題報告モーダルが開いている、**操作** "Capture Screenshot" ボタンをクリック、**期待結果** gwt ウィンドウの画像キャプチャが取得され、サムネイルとしてプレビュー表示される。画像は `~/.gwt/reports/images/` に保存される
3. **前提条件** 画像キャプチャが添付されている、**操作** Issue を送信する、**期待結果** Issue 本文にローカル画像パスが記載される（`Screenshot: ~/.gwt/reports/images/{timestamp}.png`）

---

### ユーザーストーリー 5 - 診断情報の選択的収集 (優先度: P1)

ユーザーとして、問題報告時に送信する診断情報の範囲をチェックボックスで選択し、収集内容をプレビューで確認してから送信したい。

**独立したテスト**: 報告モーダル内の診断情報チェックボックスのON/OFFに応じて、プレビュー内容が変わることを確認

**受け入れシナリオ**:

1. **前提条件** 問題報告モーダルが開いている、**操作** 診断情報エリアを確認する、**期待結果** "System Info"（OS・バージョン等）、"Application Logs"（~/.gwt/logs/ 最新N行）、"Screen Capture"（テキスト/画像）のチェックボックスが表示されている。"System Info" はデフォルトON
2. **前提条件** チェックボックスの選択を変更、**操作** "Preview" ボタンをクリック、**期待結果** 選択された情報のみがプレビューエリアに表示される。機密情報は自動マスキングされている
3. **前提条件** プレビューが表示されている、**操作** プレビュー内容を編集（テキスト部分）、**期待結果** 編集内容が送信時に反映される（ユーザーが手動で機密情報を削除可能）

---

### ユーザーストーリー 6 - 送信先リポジトリの選択 (優先度: P1)

ユーザーとして、問題報告の送信先を gwt 本体リポジトリ（akiojin/gwt）または現在作業中のリポジトリから選択したい。

**独立したテスト**: 送信先ドロップダウンで作業中リポジトリを選択し、そのリポジトリに Issue が作成されることを確認

**受け入れシナリオ**:

1. **前提条件** プロジェクトが開かれている、**操作** 報告モーダルの送信先を確認する、**期待結果** デフォルトで "akiojin/gwt" が選択されている。ドロップダウンに現在の作業リポジトリも表示される
2. **前提条件** 作業リポジトリを送信先に選択、**操作** Issue を送信する、**期待結果** 作業リポジトリに Issue が作成される

---

### ユーザーストーリー 7 - プライバシー保護（自動マスキング + プレビュー） (優先度: P1)

ユーザーとして、報告に含まれる機密情報（API キー、トークン、パスワード等）が自動的にマスキングされ、送信前にプレビューで確認できるようにしたい。

**独立したテスト**: API キーやトークンパターンを含む診断情報が自動マスキングされることを確認

**受け入れシナリオ**:

1. **前提条件** ログに `ANTHROPIC_API_KEY=sk-ant-xxx...` のような文字列が含まれる、**操作** "Application Logs" を選択してプレビューを表示、**期待結果** `sk-ant-xxx...` 部分が `[REDACTED]` にマスキングされている
2. **前提条件** プレビューが表示されている、**操作** マスキングされた内容を確認する、**期待結果** API キー、Bearer トークン、パスワードフィールド値、GitHub トークン (ghp\_\*, gho\_\*) がマスキングされている
3. **前提条件** プレビューにマスキング漏れがある、**操作** プレビューテキストを手動編集、**期待結果** 編集後の内容が送信に反映される

---

### ユーザーストーリー 8 - 送信失敗時のフォールバック (優先度: P2)

ユーザーとして、GitHub Issues への送信が失敗した場合（オフライン・認証エラー等）、報告内容をクリップボードにコピーし、ブラウザで Issue 作成ページを開いて手動で報告したい。

**独立したテスト**: GitHub API エラー時にクリップボードコピー + ブラウザ起動のフォールバックが動作することを確認

**受け入れシナリオ**:

1. **前提条件** GitHub API が到達不可（オフライン）、**操作** Issue 送信を試みる、**期待結果** エラーメッセージが表示され、"Copy to clipboard & open in browser" ボタンが表示される
2. **前提条件** フォールバックボタンが表示されている、**操作** ボタンをクリック、**期待結果** 報告内容がマークダウン形式でクリップボードにコピーされ、該当リポジトリの Issue 新規作成ページが外部ブラウザで開く
3. **前提条件** GitHub 認証が未設定、**操作** Issue 送信を試みる、**期待結果** 認証未設定を検知し、同様のフォールバック動作を行う

## エッジケース

- GitHub API レート制限に達した場合: フォールバック（クリップボード + ブラウザ）を使用
- スクリーンキャプチャ権限が未許可の場合（macOS）: 権限要求ダイアログを表示し、テキストキャプチャへの代替案を案内
- ログファイルが存在しない場合（初回起動直後等）: "Application Logs" チェックボックスをグレーアウトし "No logs available" と表示
- フォームに何も入力せず送信しようとした場合: "Title" フィールドの必須バリデーション。Submit ボタンは Title が空の間は disabled
- 複数ウィンドウが開いている場合のスクリーンショット: アクティブウィンドウ（フォームを表示しているウィンドウ）をキャプチャ
- 報告モーダルが開いている状態で新たなエラーが発生: エラーはキューに入れ、モーダルを閉じた後にトーストを表示
- プロジェクト未オープン状態で報告: 送信先は akiojin/gwt 固定。作業リポジトリ選択は不可
- 非常に長いログ出力: 最新 200 行に制限し、全文が必要な場合はローカルパスを案内

## 要件 *(必須)*

### 機能要件

#### エラーバス・構造化エラー型

- **FR-001**: 全 Tauri コマンドのエラーに severity レベルを付与する（info / warning / error / critical）
- **FR-002**: エラー応答を構造化する（severity、error_code、message、command_name、timestamp を含む構造化エラー型）
- **FR-003**: フロントエンドにグローバルエラーバスを導入し、全 invoke() 呼び出しのエラーを中央で受信する
- **FR-004**: 既存の全コンポーネントの invoke() 呼び出しをエラーバス経由に移行する（全面リファクタリング）
- **FR-005**: severity=error 以上のエラーのみ報告トーストを表示する。info/warning は既存のインライン表示を維持する

#### トースト通知

- **FR-006**: エラー検知時にトースト通知を表示し、"An error occurred. Report?" メッセージと "Report" アクションリンクを含める
- **FR-007**: 同一セッション内で同一フィンガープリント（error_code + command_name のハッシュ）のトーストは1回のみ表示する
- **FR-008**: トースト通知の "Report" リンクをクリックすると、問題報告モーダルがエラー情報をプリフィルした状態で開く

#### 報告フォーム UI

- **FR-009**: 統合報告モーダルダイアログを提供し、"Bug Report" / "Feature Request" のタブで種別を切り替える
- **FR-010**: Bug Report フォームのフィールド: Title（必須）、Steps to Reproduce、Expected Result、Actual Result
- **FR-011**: Feature Request フォームのフィールド: Title（必須）、Description、Use Case、Expected Benefit
- **FR-012**: 送信先リポジトリのドロップダウン（デフォルト: akiojin/gwt、オプション: 作業中リポジトリ）
- **FR-013**: 診断情報の選択的収集チェックボックス: "System Info"（デフォルトON）、"Application Logs"、"Screen Capture"（テキスト/画像）
- **FR-014**: "Preview" ボタンで収集情報のプレビューを表示し、プレビューテキストの手動編集を可能にする
- **FR-015**: エラー自動検知から開かれた場合、エラー情報（メッセージ・コマンド名・タイムスタンプ）をフォームにプリフィルする

#### スクリーンキャプチャ

- **FR-016**: テキストキャプチャ: 既存の Cmd+Shift+C の `collectScreenText()` を流用し、構造化テキストを取得する
- **FR-017**: *(延期)* 画像キャプチャ: Rust 側で OS ネイティブ API（macOS: CGWindowListCreateImage）を使用し、gwt ウィンドウのみをキャプチャする
- **FR-018**: *(延期)* キャプチャした画像は `~/.gwt/reports/images/{timestamp}.png` に保存する
- **FR-019**: *(延期)* 画像キャプチャはフォーム内にサムネイルプレビューとして表示する

#### プライバシー・マスキング

- **FR-020**: 診断情報・ログ・テキストキャプチャに対して、機密情報パターンを自動マスキングする
- **FR-021**: マスキング対象パターン: API キー（sk-ant-\*, sk-\*）、Bearer トークン、GitHub トークン（ghp\_\*, gho\_\*, github\_pat\_\*）、パスワードフィールド値、環境変数の秘匿値
- **FR-022**: マスキング後の内容をプレビューで表示し、ユーザーが手動編集可能にする

#### GitHub Issues 連携

- **FR-023**: gh CLI を使用して Issue を作成する。認証は gh CLI の認証情報を使用する
- **FR-024**: Bug Report には "bug" ラベル、Feature Request には "enhancement" ラベルを自動付与する
- **FR-025**: Issue 本文はテンプレート形式とする（環境情報セクション、再現手順セクション、診断情報セクション等を含む構造化マークダウン）
- **FR-026**: 画像キャプチャは Issue 本文にローカルパスを記載する（`Screenshot: ~/.gwt/reports/images/{timestamp}.png`）
- **FR-027**: 送信失敗時のフォールバック: 報告内容をクリップボードにコピーし、該当リポジトリの Issue 新規作成ページを外部ブラウザで開く

#### メニュー導線

- **FR-028**: ネイティブメニューに "Help" メニューを新規追加する
- **FR-029**: Help メニューに "Report Issue..." と "Suggest Feature..." の2項目を配置する
- **FR-030**: "Report Issue..." にキーボードショートカットを割り当てる

### 非機能要件

- **NFR-001**: エラーバスのオーバーヘッドは既存の invoke() 呼び出しに対して 5ms 以内であること
- **NFR-002**: スクリーンキャプチャ（テキスト・画像とも）は 500ms 以内に完了すること
- **NFR-003**: マスキング処理はプレビュー表示前に同期的に完了し、マスキング漏れによる機密情報の外部送信を防止すること
- **NFR-004**: 報告モーダルは開いてから 100ms 以内にレンダリングが完了すること
- **NFR-005**: macOS / Windows の両プラットフォームで OS ネイティブスクリーンキャプチャが動作すること

## 制約と仮定

- 画像キャプチャは OS ネイティブ API を使用するため、macOS ではスクリーン収録権限が必要（macOS 10.15+）
- GitHub Issues への画像の直接アップロードは行わない。画像はローカル保存しパスを記載する
- 既存の GitHub Integration 設定（SettingsPanel の GitHub タブ）の認証情報を流用する
- エラーバスの全面リファクタリングは既存コンポーネントのエラー表示動作を変更しない（既存のインライン表示は維持）
- 重複防止はセッション単位（アプリ再起動でリセット）。GitHub Issue 側の重複検知は行わない
- xterm.js の canvas 要素は OS ネイティブスクリーンショットでキャプチャ可能（WebView 内の描画として含まれる）
