# 機能仕様: Cleanup Remote Branches

**仕様ID**: `SPEC-ad1ac432`
**作成日**: 2026-02-18
**更新日**: 2026-02-21
**ステータス**: ドラフト
**カテゴリ**: GUI / Worktree Management
**依存仕様**:

- SPEC-c4e8f210（Worktree Cleanup GUI — 実装完了）

**入力**: ユーザー説明: "クリーンアップ処理でローカルWorktreeだけでなくリモートブランチも削除したい"

## 背景

- SPEC-c4e8f210 で実装された Worktree Cleanup 機能は、ローカル Worktree とローカルブランチのみを削除対象としており、リモートブランチの削除は明示的に範囲外とされていた
- 実運用では、ローカルをクリーンアップした後もリモートに不要なブランチが残り続け、手動で削除する手間が発生している
- PR がマージ済みまたはクローズ済みのブランチは安全に削除できるため、ローカルクリーンアップと同時にリモートも削除できるオプションが求められている

## 前提条件

- `gh` CLI がインストールされ、認証済みであること（PR 状態取得とリモート削除機能の有効化に必要）
- `gh` が利用不可の環境ではリモート削除関連の UI を非表示にし、従来通りのローカルのみのクリーンアップとして動作する

## ユーザーシナリオとテスト *(必須)*

### ユーザーストーリー 1 - リモートブランチ同時削除 (優先度: P0)

開発者として、ローカル Worktree のクリーンアップ時にリモートブランチも同時に削除したい。

**独立したテスト**: gh 認証済み環境で CleanupModal のトグルを ON にして safe ブランチを削除すると、ローカルとリモートの両方が削除される

**受け入れシナリオ**:

1. **前提条件** gh CLI が認証済み、**操作** Cleanup モーダルを開く、**期待結果** 「Also delete remote branches」トグルが表示される
2. **前提条件** トグルを ON にして safe なブランチを選択、**操作** "Cleanup" ボタンをクリック、**期待結果** ローカル Worktree + ローカルブランチ + リモートブランチが削除される
3. **前提条件** トグルが OFF、**操作** "Cleanup" ボタンをクリック、**期待結果** 従来通りローカルのみ削除される（リモートは残る）
4. **前提条件** gone ブランチ（リモートが既に削除済み）を選択し、トグル ON、**操作** "Cleanup" ボタンをクリック、**期待結果** ローカルのみ削除され、リモート削除はスキップされる

---

### ユーザーストーリー 2 - PR 状態の可視化と安全性判定 (優先度: P0)

開発者として、各ブランチの PR 状態（Merged / Open / Closed / None）を確認し、リモート削除の安全性を判断したい。

**独立したテスト**: PR がマージ済みのブランチに緑色の「PR: merged」バッジが表示される。PR Open のブランチはトグル ON 時に安全性が Warning に格下げされる

**受け入れシナリオ**:

1. **前提条件** PR がマージ済みのブランチが存在、**期待結果** Cleanup モーダルに「PR: merged」バッジが緑色で表示される
2. **前提条件** PR が Open のブランチが存在、**期待結果** 「PR: open」バッジがオレンジ色で表示される
3. **前提条件** PR がマージされずにクローズされたブランチが存在、**期待結果** 「PR: closed」バッジが緑色で表示される
4. **前提条件** PR が存在しないブランチ、**期待結果** PR バッジは表示されない
5. **前提条件** 「Also delete remote」トグルが ON でローカルが Safe かつ PR が Open のブランチ、**期待結果** 統合安全性レベルが Warning に格下げされる（ドットが黄色に変化）
6. **前提条件** 「Also delete remote」トグルが OFF、**期待結果** PR 状態に関わらずローカルの安全性レベルのみで判定される

---

### ユーザーストーリー 3 - 単一ブランチ削除のモーダル統合 (優先度: P1)

開発者として、コンテキストメニューの「Cleanup this branch」でも CleanupModal を経由し、リモート削除のオプションを利用したい。

**独立したテスト**: 「Cleanup this branch」を選択すると CleanupModal が開き、該当ブランチにチェックが入った状態で表示される

**受け入れシナリオ**:

1. **前提条件** ブランチを右クリック、**操作** "Cleanup this branch" を選択、**期待結果** CleanupModal が開き、該当ブランチにチェックが入った状態で表示される
2. **前提条件** モーダル内でトグル ON にして Cleanup 実行、**期待結果** ローカル + リモートが削除される

---

### ユーザーストーリー 4 - gone ブランチとリモート生存ブランチの区別 (優先度: P1)

開発者として、リモートが既に削除済み（gone）のブランチと、リモートが生きているブランチを一目で区別したい。

**独立したテスト**: トグル ON 時に gone ブランチの gone バッジが強調表示され、リモート削除不要であることが視覚的に明確である

**受け入れシナリオ**:

1. **前提条件** gone ブランチが存在、「Also delete remote」トグルが ON、**期待結果** gone バッジが強調表示され、リモート削除が不要であることが視覚的に明確である
2. **前提条件** リモートが生きているブランチが存在、**期待結果** gone バッジは表示されず、トグル ON 時にリモート削除の対象であることが分かる

---

### ユーザーストーリー 5 - リモート削除の部分失敗 (優先度: P1)

開発者として、ローカル削除は成功したがリモート削除が失敗した場合、何が起きたかを正確に把握したい。

**独立したテスト**: リモート削除が失敗しても、ローカル削除は維持され、結果ダイアログに全件の詳細が表示される

**受け入れシナリオ**:

1. **前提条件** 3ブランチを選択、うち1つのリモート削除がネットワークエラーで失敗、**操作** Cleanup 実行、**期待結果** ローカルは3つとも削除され、リモート削除の失敗が結果ダイアログに「Local: OK / Remote: Failed (reason)」のように全件表示される
2. **前提条件** 権限不足でリモート削除が失敗、**期待結果** ローカル削除は維持（ロールバックしない）、リモート失敗の理由が表示される

---

### ユーザーストーリー 6 - gh 未対応環境 (優先度: P1)

開発者として、gh CLI がない環境でもクリーンアップ機能が正常に動作してほしい。

**独立したテスト**: gh 未対応環境で CleanupModal を開くと、トグルと PR バッジが非表示になり、従来通りのローカルのみクリーンアップが利用可能

**受け入れシナリオ**:

1. **前提条件** gh CLI が未インストールまたは未認証、**操作** Cleanup モーダルを開く、**期待結果** 「Also delete remote branches」トグルが非表示。PR バッジも表示されない。従来通りのローカルのみクリーンアップが利用可能

---

### ユーザーストーリー 7 - 強制Cleanupモード（unsafe限定） (優先度: P1)

開発者として、unsafe な Worktree を明示的に force 指定で削除できることを認識しながら Cleanup を実行したい。

**独立したテスト**: CleanupModal で「Force cleanup」を ON にして unsafe ブランチを削除した場合、結果ダイアログに force 実行の注記が表示される。disabled 行は ON でも選択不可のまま。

**受け入れシナリオ**:

1. **前提条件** Cleanup モーダルを開く、**操作** トグル群を確認、**期待結果** 「Force cleanup」トグルが表示される（初期値 OFF）
2. **前提条件** unsafe ブランチを選択し「Force cleanup」を ON、**操作** "Cleanup" → 確認ダイアログで承認、**期待結果** force 指定で cleanup が実行される
3. **前提条件** protected/current/agent-running の行が存在し「Force cleanup」を ON、**操作** チェックを試みる、**期待結果** 該当行は選択不可（disabled 維持）
4. **前提条件** force 指定で cleanup 実行済み、**操作** 結果ダイアログを確認、**期待結果** force 実行であることを示す注記が表示される

## エッジケース

- protected branch（main/master/develop/release）はリモート削除の対象外（ローカルと同じく Disabled）
- gone ブランチはリモート削除をスキップ（既にリモートに存在しない）
- トグル ON + PR Open のブランチでも削除は可能（Warning 表示で注意喚起するが、ブロックはしない）
- ネットワーク切断中のリモート削除失敗はローカル削除に影響しない
- PR 状態取得がタイムアウトした場合、PR バッジは表示されず、安全性レベルは変更しない
- 同一ブランチに複数の PR（Open + Merged 等）がある場合、最新の PR 状態を採用する
- リモートは `origin` のみを対象とする（複数リモートは範囲外）
- `Force cleanup` は unsafe（warning/danger）削除時の force 実行にのみ適用し、protected/current/agent-running のガードは解除しない
- `Force cleanup` トグル状態はセッション内のみ有効とし、モーダル再オープン時は OFF に戻す

## 要件 *(必須)*

### 機能要件

- **FR-600**: アプリ起動時に gh CLI の利用可否をチェックし、結果をキャッシュしなければ**ならない**
- **FR-601**: gh CLI が利用不可の場合、リモート削除関連の全 UI（トグル、PR バッジ）を非表示にしなければ**ならない**
- **FR-602**: CleanupModal に「Also delete remote branches」グローバルトグルを提供しなければ**ならない**。トグルの状態はプロジェクト設定として永続化する
- **FR-603**: トグル ON 時、選択された全ブランチのリモートブランチも削除対象としなければ**ならない**（gone ブランチを除く）
- **FR-604**: リモート削除は GitHub API 経由で実行しなければ**ならない**
- **FR-605**: 削除順序はブランチ単位で連続（ローカル削除 → リモート削除 → 次のブランチ）としなければ**ならない**
- **FR-606**: ローカル削除が成功しリモート削除が失敗した場合、ローカル削除は維持し、リモート失敗のみを報告しなければ**ならない**
- **FR-607**: 結果ダイアログに各ブランチのローカル・リモート両方の削除結果を全件表示しなければ**ならない**
- **FR-608**: CleanupModal を開いた時点で全ブランチの PR 状態を非同期で一括取得しなければ**ならない**。取得中はスピナーを表示し、取得完了で安全性レベルと PR バッジを更新する
- **FR-609**: PR 状態（Merged / Open / Closed / None）を各ブランチ行にバッジとして表示しなければ**ならない**。Merged / Closed は緑、Open はオレンジ
- **FR-610**: 「Also delete remote」トグル ON 時、安全性レベルは統合判定としなければ**ならない**:
  - ローカルが Safe + PR Merged → Safe
  - ローカルが Safe + PR Closed → Safe
  - ローカルが Safe + PR Open → Warning に格下げ
  - ローカルが Safe + PR None → Warning に格下げ
  - ローカルが Warning 以上 → ローカルの安全性をそのまま維持（PR 状態で格下げしない）
  - トグル OFF 時は従来のローカルのみの安全性判定を使用
- **FR-611**: gone ブランチの gone バッジを強調表示し、「Also delete remote」トグル ON 時にリモート削除不要であることを視覚的に明示しなければ**ならない**
- **FR-612**: コンテキストメニューの「Cleanup this branch」は CleanupModal を開き、該当ブランチにチェックが入った状態で表示しなければ**ならない**（単体即時削除ルートを廃止し、モーダル経由に統合）
- **FR-613**: 既存の FR-508（リモートブランチは削除しては**ならない**）を、トグル OFF 時のみ適用するよう変更する
- **FR-614**: 既存の unsafe 確認ダイアログに「リモートブランチも削除されます」の警告を統合しなければ**ならない**（追加の確認ステップは設けない）
- **FR-615**: CleanupModal に「Force cleanup」グローバルトグルを提供しなければ**ならない**。トグル初期値は OFF とする
- **FR-616**: 強制実行の適用範囲は unsafe（warning/danger）削除時に限定しなければ**ならない**。safe のみ選択時は force を使っては**ならない**
- **FR-617**: `Force cleanup` が ON の場合でも、protected/current/agent-running の削除禁止ガードを維持しなければ**ならない**
- **FR-618**: force 指定で実行された cleanup は、結果ダイアログに force 実行であったことを示す注記を表示しなければ**ならない**

### 非機能要件

- **NFR-600**: PR 状態取得は CleanupModal 表示をブロックしてはならない。モーダルは即座に開き、PR 情報は非同期で取得・反映する
- **NFR-601**: gh CLI チェックはアプリ起動時に1回のみ実行し、結果をメモリにキャッシュする。モーダル表示時のオーバーヘッドをゼロにする
- **NFR-602**: リモート削除のタイムアウトはブランチあたり 10 秒とし、タイムアウト時は失敗扱いとする
- **NFR-603**: gh CLI 認証チェックのタイムアウトは 5 秒とし、タイムアウト時は利用不可扱いとする
- **NFR-604**: 自動テストが必須である。バックエンドの安全性判定・リモート削除パス・PR 状態取得、フロントエンドのトグル・バッジ・結果表示のいずれも自動テストでカバーする

## 制約と仮定

- gh CLI はユーザー環境に別途インストールされている前提（gwt がインストールする責務は持たない）
- リモートは `origin` のみを対象とする（複数リモートの対応は範囲外）
- GitHub Branch Protection ルールは参照しない（ローカルの保護リストのみで判定）
- リモート削除失敗のリトライ機構は提供しない（再度 Cleanup を実行すれば良い）
- PR 状態取得は最大 200 件の一括取得で十分とする（200 件を超える大規模リポジトリでは取得漏れを許容）
