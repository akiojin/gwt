# Report Format — Reference

## Report template

The final output of `gwt-arch-review` is a markdown report. Use this template as the structure. Adapt section depth based on findings.

```markdown
# Architecture Review Report

**Repository**: <repo name>
**Scope**: <full | crate subset | changed files>
**Date**: <YYYY-MM-DD>
**Reviewed by**: gwt-arch-review

---

## Executive Summary

<2-3 sentences summarizing overall codebase health, the single most important concern, and the recommended first action.>

## Metrics Dashboard

| Metric | Value | Status |
|--------|-------|--------|
| Total modules/crates | N | — |
| Total source files | N | — |
| Median file size | N lines | — |
| Files > 300 lines | N | ok / warn / critical |
| Files > 500 lines | N | ok / warn / critical |
| Circular dependencies | N | ok / warn / critical |
| Shallow modules (depth < 5) | N | ok / warn / critical |
| God objects (> 10 methods) | N | ok / warn / critical |
| Untested public API items | N | ok / warn / critical |
| Domain boundary violations | N | ok / warn / critical |

## Findings

### Critical

Findings that pose immediate architectural risk. Should be addressed before new feature work.

- **[C1]** <concise title>
  - **Files**: `<path1>`, `<path2>`
  - **Issue**: <what is wrong, with evidence>
  - **Impact**: <why it matters — maintainability, correctness, agent confusion>
  - **Suggested action**: <concrete next step>

### High

Findings that cause significant friction or risk but don't block current work.

- **[H1]** <title>
  - **Files**: `<paths>`
  - **Issue**: <description>
  - **Impact**: <why it matters>
  - **Suggested action**: <next step>

### Medium

Findings worth addressing during related work but not urgently.

- **[M1]** <title>
  - **Files**: `<paths>`
  - **Issue**: <description>
  - **Suggested action**: <next step>

### Low

Minor improvements for when there is spare capacity.

- **[L1]** <title>
  - **Files**: `<paths>`
  - **Suggested action**: <next step>

## Improvement Proposals

Priority-ordered, actionable proposals that bundle related findings.

### Proposal 1: <title>

- **Addresses**: [C1], [H2], [M4]
- **Estimated effort**: small / medium / large
- **Approach**: <1-3 sentences describing the recommended approach>
- **Next step**: Create improvement SPEC via `gwt-design-spec`

### Proposal 2: <title>

- ...

## Feedback Loop

To act on this review:

1. Select the top 1-3 improvement proposals based on current priorities
2. Create improvement SPECs for selected proposals using `gwt-design-spec`
3. Implement through the standard `gwt-plan-spec` → `gwt-build-spec` pipeline
4. Run `gwt-arch-review` again after implementation to verify improvement
5. Repeat weekly or bi-weekly to maintain architectural health

---

*Generated by gwt-arch-review*
```

## Severity Definitions

| Level | Definition | Action timeline |
|-------|-----------|-----------------|
| **Critical** | Active harm to correctness, maintainability, or agent comprehension. Architectural decay accelerates if unaddressed. | Address before next feature work |
| **High** | Significant friction or risk. Not blocking but accumulating cost. | Address within 1-2 sprints |
| **Medium** | Worth fixing during related work. Unlikely to cause problems in isolation. | Address opportunistically |
| **Low** | Polish and minor improvements. Nice to have. | Address when convenient |

## Severity Budget

To keep reports focused and actionable:

| Level | Target count | Maximum |
|-------|-------------|---------|
| Critical | 3-5 | 7 |
| High | 5-10 | 15 |
| Medium | 10-15 | 20 |
| Low | 5-10 | 15 |

If you find more issues than the maximum, consolidate related findings into broader items. A report with 50 findings is less useful than one with 20 well-prioritized findings.

## Finding Quality Checklist

Every finding must pass this checklist:

- [ ] **Specific**: Cites exact file paths (and line ranges when relevant)
- [ ] **Evidence-based**: Shows what was observed, not just what might be wrong
- [ ] **Impactful**: Explains why this matters (not just that it violates a rule)
- [ ] **Actionable**: Suggests a concrete next step
- [ ] **Non-redundant**: Doesn't repeat another finding with different wording

## Improvement Proposal Quality

Each proposal should:

- Bundle 2-5 related findings (don't create a proposal per finding)
- Have a clear scope boundary (what is included and excluded)
- Suggest an approach, not just "fix it"
- Estimate effort honestly (small = hours, medium = days, large = week+)
- Reference `gwt-design-spec` as the entry point for creating the improvement SPEC

## Report Anti-patterns

Avoid these:

- **Laundry list**: Listing every minor issue without prioritization
- **Academic critique**: Pointing out theoretical violations without practical impact
- **Vague findings**: "Code quality could be improved" without specifics
- **Missing evidence**: "This module is too coupled" without showing the coupling
- **Prescriptive solutions**: Designing the full solution in the report (that's `gwt-design-spec`'s job)
- **Scope creep**: Reviewing test quality, CI config, documentation quality — stay focused on architecture
