description: ユーザー要件に基づいて、仕様品質チェックリストを生成します（実装テストではありません）。
---

## ユーザー入力

```text
$ARGUMENTS
```

追加の指示があれば必ず反映します。

## チェックリストの考え方

- チェックリストは「要件（spec.md）の品質」を検証するためのものです。
- 実装が正しいか（ボタンが動くか、APIが200か等）を検証するものではありません。

## 手順概要

1. **対象SPEC IDの決定（必須）**:
   - ユーザー入力に `SPEC-[a-f0-9]{8}` が含まれる場合、それを対象SPEC IDとします。
   - 含まれない場合は `specs/specs.md` を提示し、ユーザーに対象SPEC IDを選んでもらってください（このコマンドはそこで停止）。

2. **前提確認（JSON）**:
   - リポジトリルートで以下を実行し、`FEATURE_DIR` を取得します。
     - `.specify/scripts/bash/check-prerequisites.sh --json --spec-id <SPEC_ID>`

3. **コンテキスト読込**:
   - `spec.md`（必須）
   - `plan.md` / `tasks.md`（存在する場合）

4. **意図の確認（必要な場合のみ）**:
   - ユーザー入力から、対象領域（例: UX / セキュリティ / 運用 / API / テスト方針）を特定します。
   - 重要な分岐がある場合のみ、最大3問まで短い確認質問をします。

5. **チェックリスト生成**:
   - `specs/<SPEC_ID>/checklists/` を作成します（存在しない場合）。
   - ファイル名例: `requirements.md` / `security.md` / `ux.md` など。
   - `.specify/templates/checklist-template.md` を雛形にし、サンプル項目を残さず実項目に置換します。
   - 項目は `CHK001, CHK002...` の連番で、要件の **網羅性/明確性/一貫性/測定可能性/境界条件** を点検できる内容にします。

## 出力

- 生成したチェックリストのパス
- 重点的にチェックする観点（要約）
