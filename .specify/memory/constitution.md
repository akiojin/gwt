# GWT (Git Worktree Manager) 原則

## 基本原則

### I. シンプルさの追求

- 設計・実装は複雑にせず、シンプルさの極限を追求する
- ただし、ユーザビリティと開発者体験の品質は決して妥協しない
- CLI操作の直感性と効率性を技術的複雑さより優先

### II. テストファースト（譲れない）

- TDD必須: 仕様策定 → テスト作成 → ユーザー承認 → 実装
- Spec Kit承認前およびSpec準拠のTDD完了前の実装着手を全面禁止
- 違反タスクは即時差し戻し

### III. 既存コードの尊重

- 既存ファイルのメンテナンスを無視して、新規ファイルばかり作成するのは禁止
- 既存ファイルの改修を優先する
- 大規模ファイルは適切な粒度で分割し、単一責任の原則を守る

### IV. 品質ゲート

- エラーが発生している状態で完了としない
- commitlint、markdownlint、ESLintのチェックを必ず通過させる
- テストカバレッジを維持・向上させる

### V. 自動化の徹底

- release-pleaseによるバージョン判定とリリースノート生成を100%自動化
- Conventional Commitsを例外なく遵守
- CI/CDパイプラインで品質を担保

## 技術スタック

- **言語**: TypeScript 5.8+
- **ランタイム**: Bun >= 1.0
- **CLI UI**: Ink.js 6.x（React 19ベース）
- **Web UI**: React 19 + Vite + Fastify + Tailwind CSS
- **テスト**: Vitest + Playwright
- **ログ**: Pino（構造化ログ）

## ワークフロー規約

### ブランチ戦略

- `main`: 本番コード（保護済み、リリースのみ）
- `develop`: 機能統合ブランチ
- `feature/*`: 新機能（develop ← → develop）
- `hotfix/*`: 本番修正（main ← → main）
- `bugfix/*`: バグ修正（develop ← → develop）

### コミット規約

- Conventional Commits形式を厳守（feat:/fix:/docs:/chore:等）
- 1コミット1タスク（抱き合わせ禁止）
- コミット前にcommitlint自己検証必須

## ガバナンス

- この原則はすべての開発作業に優先する
- 原則の修正には文書化、承認、移行計画が必要
- すべてのPR/レビューはこの原則への準拠を検証する
